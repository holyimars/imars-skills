# LSP collaboration protocol, and native-only fallbacks shared by both tools

## LSP: opportunistic corroboration layer, never a chain head

Claude Code's own LSP tool (`findReferences`, `goToDefinition`, `hover`, `documentSymbol`, `goToImplementation`, `prepareCallHierarchy`, `incomingCalls`/`outgoingCalls`, `workspaceSymbol`) is a THIRD source of structural information, independent of both `codebase-memory-mcp` and `codegraph`.
It is deliberately NOT placed at the head of any row in `SKILL.md`'s decision table — TypeScript now has real field data (below), but it's a genuinely mixed result, not a clean win, and Java remains structurally unavailable in this installation.

- **TypeScript LSP is now field-verified (2026-07-20, `plus-ui`) — but getting it working revealed an operational requirement not documented anywhere: a GLOBAL npm install of `typescript-language-server` is NOT sufficient.**
  `npm i -g typescript-language-server typescript` (confirmed on PATH via `which`) still produced `"Command 'typescript-language-server' not found or is in an unsafe location (current directory)"` on every LSP call. Only after additionally running `npm i -D typescript-language-server typescript` INSIDE the target repo (`plus-ui`) did the LSP tool start working. The official `typescript-lsp@claude-plugins-official` plugin's own README only documents the global-install path — that alone was not enough in this environment. **Practical guidance: if LSP calls fail with this exact "not found or unsafe location" message even after a global install, install the same packages as a local devDependency in the target repo before concluding LSP is unavailable.**
  (These test installs were reverted afterward via `npm uninstall`/`git checkout` — they were scratch verification, not intended changes to the user's real `package.json`.)
- **Once working, two of the three "expected" claims below are now confirmed — one positive, one negative — and the positive one comes with a severe caveat specific to Vue+TS stacks:**
  - **CONFIRMED POSITIVE**: `findReferences` correctly disambiguates two unrelated same-named exported functions (`getInfo` in `src/api/login.ts:97` vs `src/api/workflow/definition/index.ts:153`, no relation to each other) — each call returned only its own real caller (`store/modules/user.ts` for the first; nothing else), zero cross-contamination. This is a real, if modest, confirmation that LSP's resolution is import/type-aware, not name-only, the way `codebase-memory-mcp`'s CALLS edges are (field-verified 60% false-positive rate on `DictService.getDictLabel`'s Lombok-getter collision, see `cbm-blindspots.md`).
  - **CONFIRMED NEGATIVE, with a dangerous failure shape**: plain `typescript-language-server` has **zero support for `.vue` files** — `hover` on a `.vue` file returns a clean, explicit `"No LSP server available for file type: .vue"`. But `findReferences` called FROM a supported `.ts` file **silently omits any reference that lives inside a `.vue` file, with no error at all** — confirmed on the same `getInfo` test: the workflow-definition `getInfo`'s only real caller is in a `.vue` file (`views/workflow/processDefinition/index.vue`), and `findReferences` reported just "1 reference" (the declaration itself), giving no signal that a real caller was silently dropped. **This is a WORSE failure shape than the direct-query error** — a confident, wrong-looking-complete answer instead of an honest refusal.
  - **CONFIRMED NEGATIVE — the "LSP would close the `app.config.globalProperties` gap" hope does NOT materialize.** Tried the exact case both graph tools already miss badly (`parseTime`, 0-1/10 real recall on both tools, see the shared-blind-spots section below): `findReferences` on `parseTime`'s declaration (`src/utils/ruoyi.ts:2`) found only 7 references across 4 files — the plugin-registration line, the `ComponentCustomProperties` type-augmentation declaration, and internal re-exports — **zero of the real `proxy.parseTime(...)` call sites**, for two compounding reasons: those call sites live inside `.vue` templates (invisible per the point above), AND even the type-only reference chain through Vue's global-property type augmentation doesn't count as a "reference to this symbol" for a real compiler's language service either. **Practical conclusion: in a Vue+TS stack specifically, LSP does not close this shared blind spot — it has the exact same gap as both graph tools, just via a different mechanism (opacity to `.vue` + type-augmentation indirection instead of static-call-graph limitations).**
- **Java LSP is now confirmed STRUCTURALLY UNAVAILABLE in this Claude Code installation, not merely "not installed" — a stronger and more useful conclusion than "untested."**
  Unlike TypeScript (wired up via the official `typescript-lsp@claude-plugins-official` plugin, confirmed present and enabled in `~/.claude/settings.json`), there is no equivalent Java plugin among the installed plugins (`claude-md-management`/`feature-dev`/`frontend-design`/`playwright`/`skill-creator`/`superpowers`/`typescript-lsp` — no `java-lsp` or similar). To test whether installing a real, working server was enough on its own (mirroring what fixed the TypeScript case), installed `@vscjava/java-language-server` (npm, bin name `jdtls`, a genuine Eclipse JDT LS wrapper, ~55MB, confirmed installable) as a local devDependency in `RuoYi-Vue-Plus` — `documentSymbol` on a real `.java` file still returned `"No LSP server available for file type: .java"`, unchanged. **Conclusion: Claude Code's LSP subsystem has no known file-type-to-command mapping for `.java` at all in this installation — installing a compatible server, even correctly, cannot make it available without a corresponding official plugin (which does not currently exist here).** (This test install was reverted afterward — it had created a stray `package.json`/`package-lock.json`/`node_modules` in a pure-Maven repo that had none before; all three were deleted.)
  Do not re-attempt this without first checking whether a `java-lsp`-equivalent plugin has since been added to the marketplace — installing the server alone is a known dead end.
- **Trigger discipline (unchanged)**: only try the LSP tool once, opportunistically, when a single-symbol question is already being answered by `cg-trace.sh`/`cbm-trace.sh` — use it to corroborate a final caller count or definition location, never as the primary source, never for exploration, aggregation, or inheritance-direction questions. Given the confirmed Vue-opacity gap above, this discipline matters MORE now, not less: a clean-looking `findReferences` result on a Vue+TS repo cannot be trusted to be complete without a native-grep cross-check, exactly like the graph tools it was hoped to corroborate.
- **Failure semantics (the actual protocol)**: any error shaped like `"No LSP server available for file type: X"` or `"... not found"` means this file type has no LSP support in the current session.
  Treat it as silent absence — do not retry, do not suggest installing a language server to the user, do not let it block or delay the chain already in progress.
  The graph-tool/native-grep answer stands on its own without LSP corroboration.
- **Remaining untested claim**: being tied to the actual buffer/compiler state, LSP results should not suffer the same "index staleness" concern the graph tools have (`codegraph sync` / re-running `codebase-memory-mcp cli index_repository`) — not tested this pass (would require editing a real file and immediately re-querying); still speculation, say "expected to" / "in theory", not "confirmed", until a future pass actually tests it.
- **LSP does not exempt any mandatory native-grep fallback.**
  Even a working LSP server operates on static/compiled structure — it has no more visibility than the graph tools into `SpringUtils.getBean(X.class)` reflection lookups, MyBatis XML `namespace=` binding, or Vue/React dynamic `() => import(...)` route registration, and (per the confirmed finding above) it has NO visibility into `.vue` files at all with this specific server.
  Every native-grep MANDATORY step in `SKILL.md`'s decision table stays mandatory regardless of whether LSP was available or corroborated a result.

## Native-only: blind spots shared by BOTH `codebase-memory-mcp` and `codegraph`

These are NOT fixed by switching between the two graph tools — both were independently field-verified to have the identical gap.
Go straight to native grep/Read; do not spend a call on either graph tool first.

- **MyBatis XML mapper binding** (`namespace=` → Java Mapper interface, `<if>/<foreach>` dynamic SQL, `@Select/@Update` SQL semantics).
  Both tools index the `.xml` file as a file node but extract zero symbols from it and bind nothing to the Java interface.
  Fallback: grep the mapper interface FQN as XML `namespace=`, then Read the mapper XML directly.
  (This repo's own `*Mapper.xml` files are empty MyBatis-Plus auto-registration shells with no hand-written SQL — the blind spot is confirmed structurally but hasn't had live dynamic SQL to fail on here.)
  See `cbm-blindspots.md`'s "Java + MyBatis / MyBatis Plus" section and `codegraph-blindspots.md`'s "MyBatis XML mapper binding" section for the per-tool repro.
- **Vue/React route-level dynamic `import()`** (`() => import('@/views/x.vue')` / `lazy(() => import('@/components/x'))`).
  Both tools produce no edge from the router/registration file to the lazily-loaded component; only statically-imported components get an edge.
  Fallback: grep the component's file path/basename directly inside router config files.
  See `cbm-blindspots.md`'s "JS/TS + dynamic `import()`" section and `codegraph-blindspots.md`'s equivalent section for the per-tool repro — originally confirmed on plus-ui's Vue Router (both tools), cross-framework-confirmed for React `lazy()` on `pterodactyl/panel` (2026-07-18), confirmed a THIRD syntactic variant on 2026-07-19 (`hi.events`'s React Router v6.4+ `async lazy() { await import(...) }` data-router pattern), and re-confirmed a FOURTH time in Round 8 on a fresh `pterodactyl-panel-bench` re-clone — the React-side confirmations are codegraph-only, since cbm's indexer cannot be run on that repo at all (see the memory-usage note below). So this is a cross-framework (Vue AND React) blind spot, not a Vue-specific one.
  Round 8 also surfaced a verification trap specific to this construct: `cg-find.sh` on the component name returns BOTH the router file's lazy-import constant AND the target file — a pure same-name coincidence (the variable is conventionally named after the file), not a resolved link; `cg-impact.sh` on the same symbol shows `edgeCount: 0`. Never conclude "the graph sees this route" from a find-level hit — see `codegraph-blindspots.md`'s dynamic-`import()` section (Round 8 block).
- **Vue/React functions registered on `app.config.globalProperties`** (e.g. `app.config.globalProperties.parseTime = parseTime`, called elsewhere as `proxy.parseTime(...)` or `proxy?.parseTime(...)`).
  Field-verified as a repo-wide CLASS of blind spot, not a per-function quirk: three independently-registered functions in `plus-ui/src/plugins/index.ts` (`parseTime`, `handleTree`, `addDateRange`) were each tested separately, and all three reproduced the identical failure — both `cg-trace.sh` and `cbm-trace.sh` recalled only 0-1 out of 10-19 real call sites, because the real call goes through the injected property, which neither tool's static call-graph model resolves back to the plugin registration.
  A single native `grep -rn "proxy\.<name>\|\.<name>("` call recovered every real call site (10/10, 19/19, 10/10 across the three functions), each in under 100ms — see `references/tool-collaboration-benchmark.md` for the full per-function numbers.
  Fallback: before trusting a call-chain answer for ANY function, check whether it's registered on `app.config.globalProperties` (grep `src/plugins/index.ts` or equivalent to enumerate the full registered set) — if so, grep its real call sites directly rather than trusting either graph tool's trace.
  **This is now confirmed a THREE-way shared blind spot, not just a graph-tool gap: TypeScript LSP was specifically tested against this exact case (2026-07-20, `parseTime`) and does not close it either — see the "LSP: opportunistic corroboration layer" section above for the full result. Native grep remains the only reliable path, regardless of which of the three tools is available.**
- **Spring `getBean(runtime-computed-name)`** (e.g. `SpringUtils.getBean(grantType + "AuthStrategy")`).
  Genuinely undecidable without running the code — this is not a graph gap, no amount of static analysis resolves a value only known at request time.
  codegraph's interface/impl heuristic will honestly (if imprecisely) list every implementation as a dynamic-dispatch candidate; codebase-memory-mcp has no equivalent synthesis here at all.
  Fallback: grep the bean-name construction pattern to enumerate candidates by hand.
  See `cbm-blindspots.md`'s "Runtime bean-name lookup" section and `codegraph-blindspots.md`'s "Spring runtime bean-name lookup" section for the per-tool repro (both tested against the same `IAuthStrategy` strategy-pattern example).
- **Go: ordinary cross-package struct-field method calls** (a service/handler struct holding a field typed from ANOTHER package, e.g. `type APIV1Service struct { Store *store.Store }`, calling `s.Store.CreateMemo(...)`) — field-verified 2026-07-20 on `usememos/memos` (both tools independently confirmed a clean 0/1-or-worse miss on 4 real methods; see `codegraph-blindspots.md`'s and `cbm-blindspots.md`'s "Go" sections for the full ground truth and numbers). This is NOT an interface/impl or fluent-chain edge case — it is the single most ordinary Go dependency-injection pattern, and both tools miss it completely, a more severe gap than anything found for Java/PHP/TS so far.
  Fallback: grep the field's declared type and method name directly (`grep -rn "\.<FieldName>\.<MethodName>("`) — treat this as the PRIMARY path for Go repos with this shape, not a fallback-of-last-resort.
- **Class `extends`/inheritance direction.**
  codegraph's graph is confirmed one-directional (querying a subclass never reveals its parent, matching upstream open issue [colbymchenry/codegraph#1328](https://github.com/colbymchenry/codegraph/issues/1328); querying the parent DOES list subclasses, mislabeled under a `Called by ←` heading — don't read that as a call relationship).
  codebase-memory-mcp never modeled this relationship at all, in either direction.
  Fallback: grep the `class X extends Y` declaration line directly — this is the one relationship neither tool carries.
  See `codegraph-blindspots.md`'s "Class inheritance (`extends`)" section for the full repro (schema-level confirmation, not just text output) — codebase-memory-mcp has no equivalent section since it never modeled this relationship at all, in either direction, to have a repro against.
- **Laravel Facades, Eloquent local scopes, Eloquent relationship magic-property access, and constructor-injected interface binding** — as of 2026-07-20, cbm finally has real field data too (`hievents-bench` indexed clean, no memory issue — see the memory-usage note below for why this doesn't contradict the `pterodactyl/panel` incident), closing most of what had been a blanket "cbm unverified" gap. The picture is split by tool and by sub-case:
  - **Facades**: splits further by WHERE the facade's target lives. An APP-DEFINED facade (`Activity::event(...)` on `pterodactyl/panel`) — codegraph resolved through to the real implementation class with HIGH recall (100%, 24/24 files, corrected 2026-07-19 after a `cg-trace.sh` result-cap fix; re-verified in Round 8 on a fresh re-clone and from-scratch index, see `codegraph-blindspots.md`). The verified scope is specifically an accessor returning a concrete app-class literal (`getFacadeAccessor()` → `ActivityLogService::class`) — an interface- or string-alias-returning accessor has never been tested, so don't generalize to "Facades resolve." Real residual risk on the positive case: a global PHP function sharing the Facade method's bare name (`event()`) can produce false positives — sample-read any surprising hit. A BUILT-IN Laravel facade (`Cache::`, `Auth::`, `DB::` etc., target lives in vendor code) is a CONFIRMED CLEAN BLIND SPOT for BOTH tools (cbm: `hievents-bench`'s `Cache::remember`/`get`/`put` calls, no node to query at all; codegraph: same call sites, confirmed in the Round 7 close-out) — vendor code isn't indexed, so there's nothing to resolve to, on any tool that only indexes application code.
  - **Eloquent local scopes** (`scopeXxx` declared, invoked with the `scope` prefix dropped as `->xxx(...)`): confirmed a CLEAN miss for BOTH tools — codegraph 0/1 on `pterodactyl/panel` (re-confirmed in Round 8 on the fresh re-clone, with a new control ruling out "the call-site file wasn't indexed"), cbm 0/5 on the `spatie/laravel-permission` fixture (Round 7 — `hievents-bench` itself has zero `scopeXxx` methods to test against). Grep the un-prefixed call form; neither graph tool will find these callers.
  - **Eloquent relationship methods accessed via magic property** (`$model->relationName`, no parens): NEW 2026-07-20, tested on `hievents-bench` with cbm only — confirmed a CLEAN miss (0/1 on `Order::attendees()`), same shape and root cause as the local-scope miss (Eloquent's dynamic `__get` has no AST-visible connection to the declared relation method). codegraph was not tested against this sub-case.
  - **Constructor-injected interface binding** (standard Laravel DI: type-hint an interface, container resolves it via a `ServiceProvider` binding) — tested on `hi.events`/`hievents-bench` with BOTH tools now, and this is where the most important correction lives: for a single-owner interface method (`UserRepositoryInterface::findByIdAndAccountId`), both tools agree exactly with grep ground truth (12/12). But for a method inherited un-overridden across MANY sibling interfaces from one shared base (`RepositoryInterface::findById`, ~15 repository types), cbm found 92 direct callers (depth 1, spot-checked, consistent with grep) while codegraph — on the identical symbol — returns only 43 (its own documented impl-node union adds just 7 more, 50 total, ~54% of cbm's count). This CORRECTS the previously-published "43 real callers, cross-checked 100% against grep ground truth" claim: that cross-check was only ever run against a narrow 16-file/18-call subset, not the true ~92-method population, so "100%" overclaimed completeness. Six of cbm's extra callers were individually source-verified as genuine call sites codegraph's query simply misses — a real, currently unexplained codegraph recall gap on this specific "one method, many sibling interfaces" shape, not a cbm false-positive problem. See `cbm-blindspots.md`'s "PHP + Laravel" section and `codegraph-blindspots.md`'s "Laravel constructor-injected interface binding" section (2026-07-20 correction) for the full numbers and repro.
  - **Container STRING bindings (`app('foo')`), Blade view path-string binding, and ServiceProvider event/listener wiring are now ALL field-verified (2026-07-20, `hievents-bench`) — confirmed clean misses on BOTH tools.** See `codegraph-blindspots.md`'s and `cbm-blindspots.md`'s "Laravel — remaining untested constructs closed out" sections for the full ground truth, exact call sites, and numbers. The one residual gap: an APP-DEFINED string-keyed binding (target still in indexed source, unlike the vendor-target case tested) has no real example in any indexed repo.
  - **Methodology trap worth repeating here**: `cbm-trace.sh`'s 3rd positional arg is DEPTH (hop count), not a result LIMIT like `cg-trace.sh`'s identically-positioned 3rd arg — passing a large number expecting to raise a cap instead traces that many hops deep and silently mixes in transitive callers. Always use depth 1 when comparing against a grep-derived "direct callers" ground truth (see `cbm-blindspots.md` for the live repro).
  See `codegraph-blindspots.md`'s dedicated Facade, Eloquent-scope, and interface-binding sections, and `cbm-blindspots.md`'s "PHP + Laravel" and "`index_repository` memory usage" sections, for the full repro and citations.
- **Django / Python dynamic dispatch** (dynamic URLconf, signals) — deliberately still not field-verified against either tool, per explicit user direction to leave this out of scope. Treat as a likely-blind structural inference carried over from general framework-magic reasoning, and spot-check before relying on it for a real decision.
- **Go** is no longer untested — see the new bullet above in the shared-blind-spots list (cross-package struct-field method calls, field-verified 2026-07-20).

`SpringUtils.getBean(X.class)` — a deterministic SINGLE-target bean lookup (as opposed to the computed-name case above) — is deliberately NOT listed here as a "shared" blind spot: the two tools are asymmetric on this one.
See `SKILL.md`'s Quality rules section (the `SpringUtils.getBean(X.class)` bullet) for the canonical statement of that asymmetry and the standing mandatory grep check; see `codegraph-blindspots.md`'s dedicated section for the underlying field-verification evidence (exact call sites, 0/2 recall across three commands).
`SpringUtils.getAopProxy(this).<method>(...)` self-invocation IS listed as a genuinely shared miss (unlike the `getBean(X.class)` case above): both `cg-trace.sh`/`cg-node.sh` AND `cbm-cypher.sh dead-code-methods` independently missed the same call site in testing, so two tools "agreeing" a method looks dead does not corroborate anything here — see `codegraph-blindspots.md`'s dedicated section (and `cbm-blindspots.md`'s dead-code section) for the full repro, and grep `SpringUtils.getAopProxy(this).<method>(` as a standing check on any AOP-annotated (`@Cacheable`/`@CachePut`/`@CacheEvict`) method before trusting a dead-code verdict.

## Config-file key binding (`@Value("${key}")` ↔ `application.yml`; `import.meta.env.VITE_X` ↔ `.env`) — NOT a clean shared blind spot, partial and asymmetric

- Unlike the fully-blind items in the list above, config-binding visibility is inconsistent per script, and one assumption from earlier testing turned out wrong: `cg-find.sh` unexpectedly indexes some `application.yml` keys as `constant`-kind nodes and links them to `@Value` reference sites in Java code — a real, if incomplete, capability (field-verified 2026-07-18).
- **Profile-specific YAML (`application-dev.yml`/`application-prod.yml`) — CONFIRMED positive 2026-07-20 (RuoYi-Vue-Plus), closing the "not confirmed to extend" gap above.** `@Value("${snail-job.port}")` is defined ONLY in the profile-specific files, never the base `application.yml`; both `codegraph query` (co-resolved `spring-value`/`spring-config` nodes across BOTH profile files) and `cbm-grep.sh` (literal match in both files) correctly surface it. This partial capability is not limited to the base config file on either tool.
- `cbm-find.sh` (regex/symbol-oriented) has zero visibility into `.yml`/`.env` files at all.
  `cbm-grep.sh` (literal-text-oriented) partially does — it modeled `plus-ui/.env.development` as a Module node and found the defining line for a `VITE_APP_CLIENT_ID`-style key.
- **New asymmetry confirmed 2026-07-20 (`plus-ui`): cbm's `.env` visibility does NOT extend uniformly across variants of the same env-file family.** `VITE_APP_MONITOR_ADMIN` is defined in BOTH `.env.development:15` and `.env.production:12` (both tracked in git, neither gitignored). `cbm-grep.sh` finds the `.env.development` definition but returns NOTHING for `.env.production` — confirmed via `cbm-find.sh ".env.production"` that the file has **zero nodes of any kind** in the index, not a truncation artifact. Seeing one `.env.*` variant indexed is no evidence the others are — check each file individually.
- **`.properties` files remain genuinely untested** — checked RuoYi-Vue-Plus's 8 real `.properties` files (2026-07-20): none are `@Value`/`@PropertySource`-bound (they're i18n `MessageSource` bundles and a p6spy config, consumed via different mechanisms entirely); every extra Spring config file in that repo is YAML via a custom `PropertySourceFactory`, not `.properties`. No repo indexed by this skill has a real `@Value ↔ .properties` binding to test against.
- Net guidance: native grep of the config file directly is still faster and complete either way, and remains the only reliable path — do not write off a graph hit here as impossible, but do not trust its absence as proof the key doesn't exist either, and do not assume one `.env.*`/`.yml` variant's coverage extends to its siblings without checking.
  See `references/tool-collaboration-benchmark.md` for the full test recipe on both repos.
