# codegraph blind spots & native fallbacks (field-verified 2026-07-16, v1.4.1)

The graph is built by codegraph's own static extractor.
The constructs below either bind at RUNTIME or are handled differently than codebase-memory-mcp's graph — verified head-to-head on the same two repos (RuoYi-Vue-Plus, plus-ui) used to verify that tool's blind spots (see the sibling `cbm-blindspots.md` in this same skill).
Use the native fallback instead; do not report graph emptiness here as "no usage / dead code".

## Java interface → implementation calls — codegraph handles this BETTER than codebase-memory-mcp, but not perfectly

- The underlying `codegraph callers` command is still single-hop: querying callers of an `*Impl` class method returns exactly ONE "caller" — and that caller is the interface's own method declaration line, not a real business caller.
  Confirmed on two independent pairs: `SysDeptServiceImpl.selectDeptList` → 1 caller, `ISysDeptService.java:33` (the interface declaration itself); `SysUserServiceImpl.selectUserListByDept` → 1 caller, `ISysUserService.java:230` (same shape).
  Read naively, this looks like "this method has exactly one caller" — it does not; the real callers (`SysDeptController.list`, `.excludeChild`) are invisible from this single command.
- BUT unlike codebase-memory-mcp, codegraph's graph DOES carry an explicit interface→impl edge, and two higher-level commands surface it:
  `codegraph node <method>` labels the cross-reference as `[dynamic: interface → impl @file:line]` on the Trail section, in BOTH directions (querying the interface method shows `Calls → impl [dynamic: ...]`; querying the impl method shows `Called by ← interface [dynamic: ...]`).
  `codegraph explore` goes further and lists the blast-radius entries for BOTH the interface method (2 real callers) and the impl method (1 "caller" = the interface) in the same response, so a single `explore` call already contains the full picture.
- `codegraph impact` (multi-hop, default depth 2) also bridges this automatically: `impact` on the impl method returns the real controller methods as affected nodes, because it walks impl→interface→caller as one of its hops.
  It can still miss deeper nodes (e.g. Route nodes) at the default depth if they are more than 2 hops from the impl side — raise `-d` if a route/endpoint seems to be missing from `cg-impact.sh` output on an impl-anchored query.
- `cg-trace.sh` in this skill auto-bridges the single-hop `callers` gap: when the direct result contains a caller whose name equals the queried method's own short name (the fingerprint of an interface-declaration bridge), it re-queries callers of that bridge symbol and unions the results, flagging `bridged: true`.
  This is a heuristic, not a proof — when `bridged: true` appears, cross-check with `cg-node.sh` or `cg-explore.sh` before stating a final caller count, the same discipline the cbm-side scripts require unconditionally.
  Code-review note (2026-07-17): an earlier version of this bridge only recognized the plain `ClassName.methodName` shorthand — passing the full `qualifiedName` (`namespace::Class::method`, the field returned by `cg-find.sh`'s own `qualified_name` key) silently defeated the bridge with no warning, because the short-name extraction split on the last `.` and landed inside the package prefix instead of at the `::` separator. Fixed to split on `::` first when present; re-verified against both field-tested pairs (`SysDeptServiceImpl.selectDeptList`, `SysUserServiceImpl.selectUserListByDept`) with both name formats — both now bridge identically.
  Separately, `codegraph callers/callees/impact` report "symbol not found" via exit code 0 plus a non-JSON message on stdout (not an error exit, not stderr) — the scripts' error handling now parses stdout with `jq empty` to detect this instead of trusting the exit code, and surface it as a normal `hint` instead of crashing on a raw `jq` parse error.
- Outbound (impl → what it calls) is NOT affected — `codegraph callees` on the impl method returns its real internal calls directly, no bridging needed.

## CALLS edges are receiver/type-aware — no Lombok getter/setter name collision (field-verified 2026-07-18, direct contrast with codebase-memory-mcp)

- The sibling `cbm-blindspots.md` documents a 60% false-positive rate on `DictService.getDictLabel` callers, caused by cbm's name-only edge resolution colliding with same-named Lombok-generated getters on unrelated DTO/VO/Entity classes.
  Ran the identical scenario against codegraph: `getDictLabel` exists as a business method on `DictService`/`SysDictTypeServiceImpl` AND as a Lombok-generated getter on four unrelated classes (`SysDictData`, `SysDictDataBo`, `SysDictDataVo`, `DictDataDTO`).
  `codegraph callers "SysDictDataVo::getDictLabel"` (the Lombok getter) returned exactly its 3 real call sites, all inside `SysDictTypeServiceImpl` where that Vo's getter is genuinely invoked — zero pollution from the business method's own real external callers, and vice versa: `callers "DictService::getDictLabel"` did not include any of the Lombok-getter call sites either (its actual gap is unrelated, see the `SpringUtils.getBean(X.class)` section below).
  Root cause of the difference: codegraph's edges are keyed by qualified/receiver-typed symbol (`Class::method`), not bare method name — the collision class of bug documented for the cbm-side scripts does not reproduce here.
  This is a genuine accuracy advantage worth stating plainly, not just implied by "better than codebase-memory-mcp" language elsewhere in this file.
- **Caveat (field-verified 2026-07-18 on plus-ui): this receiver-typed guarantee covers method-level `CALLS` edges only — `cg-node.sh`'s FILE-level "used by N files" dependency aggregation is a separate heuristic and is NOT immune to name collision.**
  `cg-node.sh -f src/components/RoleSelect/index.vue` reported the file as "used by 29 files"; manual grep found ZERO genuine references to the component anywhere in the repo.
  The false 29 came from other, unrelated CRUD view files sharing generic boilerplate identifiers (`handleQuery`, `queryParams`) that this file-level aggregation associated with `RoleSelect` — a coincidental-name collision at the file-summary level, distinct from the method-level `CALLS`-edge collision this section is about.
  Fallback: never trust a bare file-level "used by N files" count for a Vue SFC as evidence of real usage either way — grep the kebab-case template tag (e.g. `<role-select`) and check for auto-import registration (`vite/plugins/components.ts` or similar) before concluding used-or-unused.
- **Extended to PHP (field-verified 2026-07-18, `pterodactyl/panel`): the receiver-typed guarantee holds on a second language, not just Java.**
  `Pterodactyl\Services\Users\UserCreationService::handle()` has exactly 4 real callers in this codebase, but a plain literal grep for the call SHAPE (`$this->creationService->handle(...)` / `$this->xxxCreationService->handle(...)`) turns up 19 hits, because this repo has a dozen-plus unrelated `*CreationService` classes (`LocationCreationService`, `NodeCreationService`, `EggVariableCreationService`, `ServerCreationService`, etc.) all injected under a similarly-named property and all calling their own same-named `handle()` method.
  `cg-trace.sh` on the exact qualified symbol returned all 4 real callers and zero of the 15 decoys, `bridged: true` (one of the 4 is itself inside another `*CreationService` class's own `handle()` method, a legitimate nested case).
  Same conclusion as the Java finding above: codegraph resolves by receiver/qualified-symbol type, so a same-named-method-across-unrelated-classes collision — which would poison a name-only tool badly here, given how common the `*CreationService->handle()` shape is in this codebase — does not reproduce.

## Spring `getBean(X.class)` — deterministic single-target lookup, complete miss (field-verified 2026-07-18, distinct from and worse than the computed-name fan-out below)

- The section below covers `SpringUtils.getBean(computedName)` selecting among MULTIPLE implementations, where codegraph responds with an honest (if imprecise) fan-out listing every candidate.
  This is a different call shape entirely: `SpringUtils.getBean(DictService.class).getDictLabel(dictType, value, separator)` — a single, statically-determinable target (there is exactly one `DictService` implementation, `SysDictTypeServiceImpl`), reached through a bean-lookup-by-`Class`-literal instead of field/constructor injection.
  Confirmed on two real call sites: `DictPatternValidator.java:51` and `ExcelDictConvert.java:65`, both calling the 3-arg `DictService.getDictLabel(String,String,String)` via `SpringUtils.getBean(DictService.class).getDictLabel(...)`.
  Neither call site appears anywhere — not in `codegraph callers "DictService::getDictLabel"` (returned only the 2 real field-injected callers, `DictTypeTranslationImpl`/`SysNoticeController`), not in `codegraph node`'s Trail section for either overload, not in `codegraph impact "DictService::getDictLabel" -d 2` (stopped at the same 2 callers + the route they lead to, 4 nodes total).
  0/2 recall on this call shape, consistent across all three commands.
  This is worse than the computed-name case: that one at least surfaces something (over-inclusive candidates, flagged as uncertain via the `[dynamic: interface → impl]` fan-out), where a `SpringUtils.getBean(X.class)` chain is invisible to the extractor outright, even though the target is fully statically determinable — unlike the computed-name case, which genuinely can't be resolved without running the code.
  Consequence for `cg-impact.sh`/blast-radius analysis is direct, not theoretical: changing `DictService.getDictLabel`'s signature or behavior would silently miss `DictPatternValidator` and `ExcelDictConvert` in the reported blast radius, a false "safe to change" read on a method with 2 more real callers than reported.
  Fallback: grep `SpringUtils.getBean(<Interface>.class)` for the interface in question directly; this pattern is common enough in Spring codebases (validators, converters, static utility classes reaching into the Spring context outside normal DI) to be worth a standing spot-check on any interface method's blast radius before reporting a final caller count.

## Spring AOP self-invocation via `SpringUtils.getAopProxy(this)` — a further complete miss, sibling to the `getBean(X.class)` gap above (field-verified 2026-07-18)

- Distinct call shape from both cases above: `SpringUtils.getAopProxy(this).selectDictTypeByType(dictType)` inside `SysDictTypeServiceImpl` itself — a self-invocation through the Spring AOP proxy, used specifically so the call routes through the proxy and the method's `@Cacheable` annotation actually takes effect (a direct `this.selectDictTypeByType(...)` call would bypass the proxy and skip caching).
  Found while cross-verifying a `cbm-cypher.sh dead-code-methods` candidate that turned out to be very much alive.
- `codegraph callers "SysDictTypeServiceImpl::selectDictTypeByType"` (via `cg-trace.sh`) and `codegraph node` (via `cg-node.sh`) were both run directly against this method — neither surfaced the `getAopProxy` call site; both looked structurally identical to a genuinely-dead method.
- This is NOT fixable by cross-checking with codebase-memory-mcp: its `dead-code-methods` template (the only dead-code capability either tool has) independently missed the same call site — see `cbm-blindspots.md`'s dead-code section for that side.
  Two tools "agreeing" a method is dead is worthless here, since both share the identical blind spot; only a native grep for `SpringUtils.getAopProxy(this).<method>(` (or a broader textual search for the method name) catches it.
- Fallback: before trusting any dead-code verdict on a method carrying `@Cacheable`/`@CachePut`/`@CacheEvict` or another AOP-relevant annotation, grep `SpringUtils.getAopProxy(this).<method>(` for that method as a standing check, same discipline as the `SpringUtils.getBean(X.class)` spot-check above.

## Spring runtime bean-name lookup (`SpringUtils.getBean(computedName)`)

- Confirmed on RuoYi-Vue-Plus's `IAuthStrategy` strategy pattern (5 implementations: Email/Password/Sms/Social/Xcx AuthStrategy, each registered under a computed bean name, selected via `SpringUtils.getBean(loginType + IAuthStrategy.BASE_NAME)` at runtime).
- codegraph applies the SAME interface/impl heuristic used for ordinary DI here: `codegraph node "IAuthStrategy.login"` lists all 5 implementations as `[dynamic: interface → impl]` candidates on the Trail.
- This is honest and useful (it correctly enumerates "one of these 5 executes, decided at runtime") but it does NOT know this is a runtime-string-computed selection specifically — it would produce the exact same fan-out for an ordinary single-bean interface.
  Do not read "5 candidates listed" as "all 5 are called on every request" or as "codegraph resolved which one is called" — it did neither; it only enumerated what implements the interface.
- Fallback for the precise selection logic: grep the bean-name construction (`+ IAuthStrategy.BASE_NAME`, `@Service("...")`) directly to see how the runtime string maps to a concrete impl.

## `cg-trace.sh` silently capped `callers`/`callees` at 20 with no truncation signal — root-caused and fixed (2026-07-19), and it invalidated two earlier findings

- **Discovery context**: while probing a Laravel repository's dependency-injected interface pattern (`hi.events`, see the interface-binding section below), a query that should have matched 43 known real callers (independently confirmed by grep) returned only 20 through `cg-trace.sh`. That number — exactly 20 — turned out not to be a coincidence.
- **Root cause**: `codegraph callers`/`codegraph callees` (the raw CLI, `codegraph callers --help`) default their own `-l`/`--limit` flag to 20, and their JSON response has no `total`/`hasMore` field of any kind — a symbol with 21+ real callers is byte-for-byte indistinguishable from one with exactly 20. `cg-trace.sh`, this skill's own wrapper script, called both commands with no `-l` at all, silently inheriting that cap on every single invocation since the script was written.
- **Verification**: re-running the SAME query with `-l 200` returned 43 callers, not 20 — confirmed genuine (not a fluke of the higher limit padding in noise) by cross-referencing the file list against independent grep ground truth: all 43 were real, on-topic call sites.
- **Fix shipped**: `cg-trace.sh` now defaults both `fetch_callers`/`fetch_callees` to `-l 200` (was: no `-l` at all, i.e. the CLI's own default of 20), accepts an optional 3rd positional arg to raise the limit further, and returns a new `possiblyTruncated: true` field (plus a hint) whenever a fetch returns exactly the limit in use — the only observable signal available, since the underlying CLI still has no total-count field.
- **Consequence — two earlier findings in this file were measurement artifacts of this exact bug, not real capability gaps**: the "Laravel Facade resolution ~29% recall" finding and the "React hook call-site recall ~67%" finding (both dated 2026-07-18) were both produced by a bare `cg-trace.sh ... in` call whose reported count happened to land at or near 20 — both are corrected below, in place, rather than left standing alongside the correction, to avoid a stale wrong number being read out of context later.
- **Standing discipline going forward**: treat any `callers`/`callees` count that is suspiciously round (20, 50, 100, 200 — whatever limit was actually used) as a signal to check `possiblyTruncated` before citing it as a final number, on this skill's own scripts just as much as on the underlying tools' raw output. This is now also stated in `SKILL.md`'s Quality rules and Token-discipline sections.

## Laravel Facade resolution — CORRECTED 2026-07-19: high real recall once the query limit is fixed, plus a genuine but narrower false-positive risk (supersedes the 2026-07-18 "~29%" finding below)

- **This entire finding was re-measured after discovering `cg-trace.sh` silently capped `callers`/`callees` at 20 results with no truncation signal (see the "`cg-trace.sh` silently capped..." section above) — the original "7/24 files, ~29% recall" verdict was an artifact of that cap, not a real capability limit.** Re-run with the fixed script (now defaulting to `-l 200`) on the identical `pterodactyl/panel` test case below.
- Test case (unchanged): `Pterodactyl\Facades\Activity` is an app-defined Facade (`getFacadeAccessor()` returns `ActivityLogService::class`), called as `Activity::event(...)` at 58 real call sites across 24 files.
- `cg-trace.sh "Pterodactyl\Services\Activity::ActivityLogService::event" in` now returns 75 caller records across 28 files, `possiblyTruncated: false`. Cross-checked file-by-file against a fresh grep of `Activity::event(`: **all 24 real ground-truth files are present — 24/24 = 100% file recall**, not 29%.
- **But 4 of the 28 returned files are genuine false positives, not more real recall**: `ForgotPasswordController.php`, `ServerObserver.php`, `SubuserObserver.php`, `UserObserver.php` contain ZERO references to `Activity`/the Facade at all — they call Laravel's global `event(new SomeEvent(...))` HELPER FUNCTION (dispatches a Laravel event, unrelated to the Facade), which codegraph's PHP extractor conflates with the `ActivityLogService::event()` METHOD purely because both are named `event`. See the new Quality-rules bullet in `SKILL.md` for the general shape of this gap (global function vs. same-named method) — it is PHP-specific (Java has no free-floating functions to collide with a method name) and narrower than the old "partial recall" framing suggested.
- **Practical guidance, revised**: a `cg-trace.sh` query against the real class behind a Facade now deserves real trust for file-level recall once you confirm `possiblyTruncated: false` (raise the limit if it fires) — but still sample-read any hit whose call site doesn't obviously reference the Facade/service, because a same-named global function is a real, confirmed way to get a false positive here. Don't skip the grep entirely (it's still the fastest way to catch a false positive like the 4 files above), but the previous advice to treat codegraph's Facade recall as merely "partial corroboration" undersold it — this is now a genuinely reliable first pass.
- **Round 8 re-verification (fresh `pterodactyl-panel-bench` re-clone and re-index, after Round 5 deleted the original)**: `cg-trace.sh` inbound on `ActivityLogService::event` again returns the full real caller population; spot-verified `AccountController::updateEmail` via direct Read — `Activity::event('user:account.email-changed')` at `app/Http/Controllers/Api/Client/AccountController.php:58` is a genuine Facade-form call site, reached only through the static `Activity::` proxy. The positive result survives a from-scratch clone+index, not just a cached index.
  Two qualifiers worth stating precisely so this finding doesn't get over-generalized:
  1. **Scope of the positive**: everything verified here is an APP-DEFINED Facade whose `getFacadeAccessor()` returns a concrete class literal (`ActivityLogService::class`) living in indexed `app/` code. A Facade whose accessor returns an interface or a string container alias has never been tested in this configuration, and built-in Illuminate Facades (`Cache::`/`Auth::`/`DB::` — accessor target in vendor code) are a separately-confirmed clean miss for both tools (see the "Laravel — remaining untested constructs closed out" section). "codegraph penetrates Facades" is not the finding; "codegraph penetrates THIS accessor shape" is.
  2. **Line-number granularity**: the line each caller record reports is the START line of the calling METHOD, not the line where `Activity::event(...)` actually appears — confirmed by Read on `updateEmail` (declaration line reported; the call sits at line 58 inside it). Method-level granularity, not statement-level. Harmless once known, but a naive cross-check of a reported line against grep output will "mismatch" on every single caller and can be misread as a false positive if this isn't kept in mind.

## Eloquent local scope invocation — confirmed clean blind spot, a distinct PHP-specific name-mangling gap (field-verified 2026-07-18, `pterodactyl/panel`)

- Laravel's Eloquent local scopes are declared as `scopeXxx(Builder $builder, ...)` but INVOKED with the `scope` prefix dropped: `$model->xxx(...)` or `Model::xxx(...)`, not `->scopeXxx(...)`.
- Test case: `HasRealtimeIdentifier::scopeWhereIdentifier` has exactly ONE real call site in this codebase, `SubstituteClientBindings.php:21` (`$builder->whereIdentifier($value)`).
- `codegraph query "whereIdentifier"` only matched the method's own declaration line and a PHPDoc `@method` annotation line — no node/edge anywhere pointed at the real call site. `cg-trace.sh ... in` on the declared symbol: **0 callers, 0/1 recall.**
- Root cause: codegraph indexes the method under its DECLARED name (`scopeWhereIdentifier`); the real invocation uses a different, un-prefixed identifier (`whereIdentifier`) that the extractor has no Eloquent-specific rule to connect back to the declaration.
- Fallback: for any Eloquent local scope method (`scopeXxx` in a Model or trait), grep the UN-PREFIXED call form (`->xxx(` / `::xxx(`) directly — querying the graph tool for the declared `scopeXxx` name will not find real callers, this is not a recall-percentage issue like the Facade case above, it is a complete miss on the one case tested.
- **Round 8 re-confirmation (fresh `pterodactyl-panel-bench` re-clone and re-index), with a control check the original probe lacked**: `cg-trace.sh` on `scopeWhereIdentifier` again returns `callers: []`, and a fuzzy `cg-find.sh -k method 'whereIdentifier'` returns only the `scopeWhereIdentifier` DECLARATIONS themselves — zero call-site nodes. The new control: the file holding the one real call site (`SubstituteClientBindings.php:21`) is itself indexed — its other class nodes are queryable normally — so this is confirmed to be a missing-EDGE gap on an otherwise-indexed file, not a coarser "this file never made it into the index" failure. Combined with cbm's own confirmed miss on the same construct (`spatie/laravel-permission` fixture, see `cbm-blindspots.md`), both tools now have this blind spot on record with the file-level alternative explanation ruled out.

## Laravel constructor-injected interface binding — a genuine capability, does NOT need the ServiceProvider binding understood at all (field-verified 2026-07-19, `hi.events`)

- This is a DIFFERENT Laravel "magic" mechanism from the Facade case above: ordinary dependency injection of an interface type (`public function __construct(private readonly EventRepositoryInterface $eventRepository)`), where Laravel's container resolves the interface to a concrete class (`EventRepository`) via a binding registered in a `ServiceProvider` (`EventRepositoryInterface::class => EventRepository::class`) — nothing Facade-specific about it, this is the standard Laravel IoC pattern used throughout any non-trivial Laravel codebase.
- Test case: `EventRepositoryInterface::findById()` is declared on the interface and implemented via inheritance (`EventRepository extends BaseRepository`, which defines `findById()`; `EventRepository` itself does not override it). Ground truth (grep): 109 total `->findById(` call sites across ~15 unrelated repository types; 18 of those specifically go through an `eventRepository`-typed property/variable.
- `cg-trace.sh "HiEvents\Repository\Interfaces::RepositoryInterface::findById" in` (querying the INTERFACE method, not the concrete class) returned, after fixing the result-cap bug above, 43 real callers spanning every real `Application/Handlers` and `Domain/Services` call site that injects ANY repository interface sharing this method name — cross-referenced against the 16-file/18-call `eventRepository`-specific ground truth, 100% were present.
- **Why this works without any special-casing for the container binding**: in PHP, a call through an interface-typed property/parameter attaches its `CALLS` edge to the INTERFACE's own declared method node — this is the same interface-typed-call behavior already documented for Java in this file's first section. Since Laravel's convention is to type-hint the INTERFACE (not the concrete class) at injection sites, querying the interface method directly surfaces the real business callers without codegraph ever needing to parse the `ServiceProvider`'s `bind()` call. The concrete method (`BaseRepository::findById`, reached when code is typed to the concrete class instead, e.g. inside `Jobs`/`Listeners` in this codebase) is a SEPARATE node with its own smaller caller set (7 in this repo) — querying only one of the two nodes misses whichever call sites use the other typing style, so both should be queried and unioned for a complete picture, same discipline as the Java interface/impl bridge.
- Practical takeaway: Laravel's constructor-injected interface pattern is NOT a blind spot at all once (a) the query targets the interface method (matching how the codebase actually types its dependencies) and (b) the result-cap bug above is accounted for. This is a materially different, more positive verdict than the Facade case's residual false-positive risk — no global-function collision risk here, since interface methods don't share PHP's global function namespace.
- **Correction (2026-07-20), significant: the "100%" cross-check above was only ever performed against the narrow 16-file/18-call `eventRepository`-specific slice, not the full ~15-interface population sharing this method — read on its own, the 43 number substantially UNDER-counts real callers, and cbm (once finally indexed on this same repo, see `cbm-blindspots.md`'s "PHP + Laravel" section) found 92 direct callers on the identical interface node, depth 1.** Even unioning this section's own documented impl-node caller set (7) with the interface node's 43 reaches only 50 — about 54% of cbm's 92. Six of cbm's extra callers (`GetEventAction`, `MessagePendingReviewJob`, `OrderCancelService`, `IsAuthorizedService`, `ResendOrderConfirmationAction`, `SendWaitlistConfirmationEmailJob`) were individually confirmed via direct source read to be genuine `->findById(...)` call sites codegraph's callers query simply never surfaces — not cbm false positives. Re-confirmed live that 43/7 are stable, unchanged numbers (not a staleness artifact of this correction) — the gap is a real, currently unexplained codegraph extraction shortfall specific to this "one method, inherited un-overridden across many sibling interfaces" shape, not a fluke of the original test. The "practical takeaway" line above ("NOT a blind spot at all") was true for the SINGLE-interface case (re-confirmed: `UserRepositoryInterface::findByIdAndAccountId`, 12/12 on both tools, see `cbm-blindspots.md`) but overclaimed for the shared-base-method case — treat this pattern's completeness as unverified for codegraph specifically until the extraction gap is root-caused, and prefer cbm's count if both tools are available.
- **Root cause pinned down (2026-07-20, same-day follow-up, multi-angle re-verification specifically aimed at answering "is this a `cg-trace.sh` script bug"): it is NOT a wrapper-script bug — it is a precisely-characterized `codegraph` core extraction limitation.**
  First ruled out the wrapper: called the raw `codegraph callers "HiEvents\Repository\Interfaces::RepositoryInterface::findById" -j -l 500` CLI directly, bypassing `cg-trace.sh`'s bridging/dedup/hint logic entirely — reproduced the identical 43 callers (and 7 on the impl node), so the wrapper is not silently dropping or truncating anything; `cg-trace.sh` faithfully reports what codegraph's own indexer resolved.
  Then diffed cbm's 92-caller list against codegraph's 43+7 by normalized `ClassName.method` key (cross-checked with `comm`; first attempt falsely showed ZERO overlap because `codegraph`'s CLI output has CRLF line endings on Windows while cbm's does not — a `tr -d '\r'` normalization fixed the false 0% overlap, worth flagging as its own methodology trap for anyone re-running this kind of cross-tool diff on Windows). After normalization: codegraph's 50 are a strict subset of cbm's 92 (0 codegraph-only entries), leaving exactly 42 codegraph misses.
  **The mechanism**: codegraph resolves a call only when the receiver is a SIMPLE expression on the same line — `$this->property->findById(...)` or `$localVar->findById(...)` — never when the receiver is itself the return value of a preceding call in a fluent chain, e.g. `$this->productRepository->loadRelation(ProductPriceDomainObject::class)->findById(...)`, even though `loadRelation()`'s return type is statically declared as the same repository interface and is in principle fully resolvable.
  Verified EXHAUSTIVELY across the entire repo, not by sampling a few cases: classified all 109 real `->findById(` call sites as DIRECT (58: receiver and `findById(` on the same source line) or CHAINED (51: the line is a bare `->findById(...)` continuing a fluent call started on an earlier line), mapped every call site to its enclosing method via nearest-preceding-`function`-declaration lookup, then cross-tabulated against codegraph's actual found (50) vs missed (42) sets — **100% clean correlation, zero exceptions in either direction**: every method codegraph finds has at least one DIRECT call site; every method it misses has ONLY chained call sites. `WebhookDispatchService` alone demonstrates this within a single file/class: its `dispatchEventWebhook` (direct `$this->eventRepository->findById($eventId)`) is found, while its other 4 methods (`dispatchAttendeeWebhook`/`dispatchCheckInWebhook`/`dispatchProductWebhook`/`dispatchOrderWebhook`, each ending a `->loadRelation(...)->findById(...)` chain of 1-3 `loadRelation` calls) are all missed — same class, same file, same method shape otherwise, chain-vs-direct is the only variable.
  **This corrects a specific wrong claim in this same correction paragraph's cross-reference**: `cbm-blindspots.md` had cited `EditProductHandler::validateChangeInProductType` as a counterexample proving "fluent-chain length alone" was ruled out as the explanation. Direct source re-read shows this was a mistake — that method's only call site (`$this->productRepository->loadRelation(ProductPriceDomainObject::class)->findById($productsData->product_id)`, `EditProductHandler.php:211-213`) is itself chained, and it IS in fact one of the 42 misses; there was no real counterexample, the earlier check must have looked at a neighboring method in the same class (`handle`/`updateProduct`, both of which DO have a separate direct call and ARE found) by mistake.
  **Secondary nuance, not chased further**: which of the two codegraph nodes (interface's 43 vs impl's 7) a direct call attaches to appears to track whether the receiver is a `$this->`-typed constructor-promoted property (→ interface node) or a bare local variable (→ impl/concrete node) — including one call reached through a `match()`-resolved container lookup with no single static type (`IsAuthorizedService::isActionAuthorized`, `$repository = match($entityType) { ... => $this->app->make(XRepositoryInterface::class) ... }; $repository->findById($entityId)`) that still resolves to the impl node — suggesting this local-variable path binds by name to whichever indexed node matches, rather than doing real type verification. Noted here for anyone re-investigating the interface/impl split specifically; does not change the chain-vs-direct finding above.
  Practical guidance updated: prefer cbm over codegraph specifically when the query symbol may be reached through a fluent builder chain (any Laravel repository/query-builder pattern with `->with(...)`/`->loadRelation(...)`/`->when(...)`-style intermediate calls before the final method) — this is now a known, mechanistically-understood gap, not just an unexplained number mismatch.

## Go (`usememos/memos`, ~803 files, field-verified 2026-07-20) — a NEW, severe blind spot: ordinary cross-package struct-field method calls are invisible, not just interface/chain edge cases

This is the first Go-language data point for this skill (previously entirely untested, lowest-priority in the target stack). The finding here is more severe than anything found for Java/PHP/TS: this is not an interface/impl split, not a fluent chain, not a runtime-computed name — it is the single most ordinary Go dependency pattern (a service struct holding a field typed from ANOTHER PACKAGE, calling a method on it), and it does not resolve at all.

- **Ground truth**: `store.Driver` (`store/driver.go:10`) is an interface with ~48 methods, implemented by three concrete `*DB` structs (`store/db/{sqlite,mysql,postgres}/*.go`) — Go's structural typing means none of these structs write `implements Driver` anywhere, satisfaction is purely by method signature match. `store.Store` (`store/memo.go`) is a wrapper struct holding a `Driver`-typed field, exposing methods like `Store.CreateMemo` that internally call `s.driver.CreateMemo(...)`. Business code holds a field `APIV1Service.Store *store.Store` (declared `server/router/api/v1/v1.go:34`, a DIFFERENT package than the caller) and calls e.g. `s.Store.CreateMemo(ctx, create)` — an entirely ordinary, single-line, non-chained call.
- **Test 1 — `Store.CreateMemo`**: real ground truth (grep) is exactly 1 caller, `server/router/api/v1/memo_service.go:115`. `codegraph callers "Store::CreateMemo" -l 50` → returned only `TestFrontendService_SitemapXML` (`frontend_test.go:176`), an unrelated test, and MISSED the real caller entirely. `codegraph callers "Driver::CreateMemo" -l 50` (the interface method) → `[]`, zero callers.
- **Test 2, 3 — `Store.UpdateMemo`/`Store.DeleteMemo`**: real ground truth is 2 callers each (`memo_service.go:496`+`memo_update_helpers.go:18` for Update; `memo_service.go:574,580` for Delete). `codegraph callers` on both → `[]`, 0/2 recall on both.
- **Test 4 — `Store.ListMemos`**: real ground truth is 8 real cross-package callers (`memo_batch_helpers.go`, `memo_service.go` ×2, `memo_service_comments.go`, `memo_service_converter.go`, `user_service_stats.go` ×2, `rss.go` ×2). `codegraph callers "Store::ListMemos"` → found exactly ONE caller, `GetMemo` at `store/memo.go:120` — a call from WITHIN the same `store` package. **Zero of the 8 real cross-package callers were found; the one hit it did find is same-package.**
- **Mechanism (consistent across all 4 methods tested)**: codegraph's Go extractor appears to resolve calls within a package reasonably (the `GetMemo`→`ListMemos` same-package hit), but a method call through a struct field whose declared type lives in a DIFFERENT package is invisible — this is a plain, direct, single-line call (`s.Store.CreateMemo(...)`), not a fluent chain, so this is a DIFFERENT and more basic root cause than the PHP chain-tail gap documented above.
- **No Go interface→impl synthesis either**: unlike Java (`[dynamic: interface → impl]` on `node`/`explore`) and Laravel's interface-typed-injection handling, `codegraph node "Driver"` (or `"store::Driver"`) shows no Trail/dynamic-dispatch section connecting the interface to its three real `*DB` implementations at all in the truncated output reached — consistent with the cross-package resolution gap above, and architecturally unsurprising since Go's structural satisfaction gives the extractor no `implements`/type-hint keyword to anchor a heuristic on the way Java/PHP's explicit interface typing does.
- **Practical guidance**: for Go repos with the (extremely common) pattern of a service/handler struct holding a field typed from another package and calling its methods, do not trust `codegraph callers`/`impact` at all — grep the field's declared type and its method name directly (`grep -rn "\.<FieldName>\.<MethodName>("`) as the primary path, not a fallback-of-last-resort the way it is for the already-well-covered Java/PHP shapes above.

## `cg-find.sh -k <kind>` with an empty pattern = accurate exhaustive listing (field-verified 2026-07-17) — routes is NOT actually a gap

- `codegraph query '' -k route -j -l 500` on RuoYi-Vue-Plus returned exactly 303 results — an EXACT match against `codegraph status -j`'s `nodesByKind.route` (303). Same exact-match verification repeated for `-k class` (482/482). On plus-ui (TS/Vue), `-k component` correctly enumerated all 99 Vue components (matches `nodesByKind.component`).
- Each route result includes the real HTTP verb and full path as its `name`, e.g. `"DELETE /auth/unlock/{socialId}"` — arguably more precise than `cbm-cypher.sh routes`, whose Cypher template only returns the path (verb not modeled in that graph schema).
- This means the original "(no direct equivalent)" verdict for "routes list" in `SKILL.md` was wrong — corrected. Dead-code and hubs are still genuine gaps (no node property encodes call-degree or callers-count for a direct filter/sort).
- Two bugs found and fixed while confirming this:
  1. `cg-find.sh -k route` (pattern omitted entirely, the natural way to ask "list all routes") crashed with `line 6: $1: unbound variable` — `set -u` treats a missing positional param as unbound, and the script assumed a pattern was always given. Fixed: `Q="${1:-}"`.
  2. Once that crash was fixed, the *default* limit (20) silently under-returned: with an EMPTY pattern, codegraph's own `-l` behaves like a per-file/group multiplier, not a literal cap — `-l 1` returned 5 rows, `-l 3` returned 15, `-l 5` returned 25 (a consistent ~5x on this repo; the exact ratio is not something to depend on). `cg-find.sh` now defaults `-l` to 500 specifically when the pattern is empty (kept at 20 for normal fuzzy search), so "list all X" actually returns everything by default. With a non-empty pattern `-l` was confirmed to behave as a normal literal cap (`-l 3` on `"system"` → exactly 3 rows) — the quirk is specific to the empty-pattern case.

## Windows: concurrent access hard-fails a full `index` rebuild — broader than upstream issue #1325 (field-verified 2026-07-18)

- Upstream issue [colbymchenry/codegraph#1325](https://github.com/colbymchenry/codegraph/issues/1325) reports `codegraph index` failing with "database file is in use" specifically when an MCP server is still running against the same project.
  Reproduced a broader version with no MCP server involved at all: launched a plain `codegraph query "Sys" --limit 5 --json` in the background, then immediately ran `codegraph index --quiet` in the foreground on the same project (local NTFS disk, WAL journal mode — none of the network-share/WSL2 caveats already documented upstream apply here).
  Result: hard failure, not a queue/retry — `[ERR] Failed to index: Could not rebuild the index — the database file is in use (EPERM, Permission denied: ...codegraph.db ...)`.
  `codegraph status` immediately after confirmed the existing index was untouched (same file/node/edge counts as before the attempt) — the failure aborts cleanly, it does not corrupt the database.
  Practical implication for this skill (CLI-only, no MCP server, so the upstream issue's specific trigger doesn't apply): a full `codegraph index --force` should not be run from one shell while another `codegraph` command — even a plain read-only `query` — might still be in flight in the same session.
  The lighter-weight `codegraph sync` used for normal incremental updates (see Quality rules) was not observed to hit this in the same test and is the safer default; reserve `index --force` for cases that actually need a full rebuild, and run it standalone.

## `install.sh` fails immediately under Git Bash / MINGW64 on Windows — root cause confirmed by reading the script (field-verified 2026-07-18)

- Open upstream issue [colbymchenry/codegraph#1294](https://github.com/colbymchenry/codegraph/issues/1294), "codegraph: unsupported OS 'MINGW64_NT-10.0-26200'", filed 2026-07-15, no maintainer response as of this pass.
  Fetched `install.sh` directly: its OS detection is `os="$(uname -s)"` followed by a `case` statement with exactly two branches, `Darwin` and `Linux` — every other value, including any MINGW/MSYS/CYGWIN string Git Bash reports, falls through to `*) echo "codegraph: unsupported OS '$os'."; exit 1`.
  `uname -s` on the machine used for this whole verification pass returns `MINGW64_NT-10.0-26200` — the exact string in the open issue, not a coincidence; running `install.sh` from Git Bash on this or any similarly-configured Windows machine will hit that `exit 1` branch every time.
  Not a blocker for this skill in practice: `README.md`'s own install instructions already use `npm i -g @colbymchenry/codegraph` (confirmed working, v1.4.1, this entire verification pass ran through it), which has no OS-detection step at all.
  Documented here so nobody "fixes" a working npm-based setup by switching to `install.sh` on a Windows/Git-Bash machine expecting it to be equivalent — it will not run at all there; use PowerShell's `install.ps1` instead if a standalone binary (not npm) is specifically wanted on Windows.
