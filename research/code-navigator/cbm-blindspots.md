# Framework blind spots & native fallbacks

The graph is built by tree-sitter + Hybrid LSP static analysis.
The constructs below bind at RUNTIME or in non-code files — the graph cannot see them.
Use the native fallback instead; do not report graph emptiness here as "no usage / dead code".

## Java interface → implementation calls (field-verified 2026-07-16, MANDATORY protocol — not an edge case)
- Blind: a call through an interface-typed variable (`@Autowired`/`@Resource`/constructor-injected field, a method param, ANY variable whose static type is the interface) attaches its `CALLS` edge to the **interface method node**, never to the implementation method node — because tree-sitter has no runtime type info and resolves by static/declared type only.
  This is **not limited to multi-impl interfaces with `@Primary`/`@Qualifier`** — it reproduces on a plain single-implementation `IFooService` → `FooServiceImpl` pair, i.e. the single most common pattern in a layered Spring codebase.
- Confirmed by direct A/B query on RuoYi-Vue-Plus (v0.9.0, field-verified, reproduced twice with different interface/impl pairs).
  `cbm-trace.sh` on `ISysDeptService.selectDeptList` (interface method) → 2 real callers (`SysDeptController.list`, `.excludeChild`).
  `cbm-trace.sh` on `SysDeptServiceImpl.selectDeptList` (the exact same logical method, impl side) → **0 callers**, no error, no warning.
  Repeated with `ISysUserService.selectUserListByDept` / `SysUserServiceImpl.selectUserListByDept` — identical result (1 real caller on the interface node, 0 on the impl node).
- **Consequence for dead-code detection specifically**: any Cypher/query that looks for "Method nodes with no inbound CALLS" will flag EVERY implementation method of EVERY interface-implementing service class as dead code, even when it is the single most-called method in the codebase.
  This is a false-positive generator, not a rare corner case, in any interface-heavy Spring/Java repo.
- Why this can't be fixed in the query itself: the obvious fix (walk `Class -[:INHERITS]-> Interface -[:DEFINES_METHOD]-> Method` and OR the two calls together) requires a 2-hop nested `EXISTS`, which this CLI's Cypher engine rejects (`unsupported EXISTS pattern — only the single-hop form '(var)-[:TYPE]->()' is supported`, tested directly).
  There is no query-side workaround.
- **Mandatory fallback (protocol, not optional)**: before reporting ANY caller count, impact result, or dead-code verdict for a method defined in a class name ending in `Impl` (or any class you know `implements` an interface), ALSO run the same query against the interface's copy of that method (same short name, defined in the `I*`/interface file) and take the union.
  A "0 callers" result on an Impl-suffixed class is not evidence of anything by itself.
- Outbound direction (impl → what it calls) is NOT affected — only inbound/callers/impact queries anchored on the impl method are blind.

## Java + MyBatis / MyBatis Plus
- Blind: XML mapper binding (`namespace` + statement id → Mapper interface method), `<if>/<foreach>` dynamic SQL, `@Select/@Update` SQL semantics.
- Fallback: Grep the mapper interface FQN as XML `namespace=`, then Read the mapper XML directly.
  For "which SQL does method X run": grep the method name inside `src/main/resources/**/ *Mapper.xml`.
- Note: LambdaQueryWrapper call chains DO resolve in the graph; only the SQL/XML semantics behind them are invisible.
- Field-tested on RuoYi-Vue-Plus (2026-07-16): this repo's `*Mapper.xml` files are empty namespace shells (MyBatis-Plus auto-registration only) — zero hand-written `<select>`/`<if>`/`<foreach>` statements found anywhere in the tree.
  On a pure MyBatis-Plus repo like this one, this blind spot simply does not trigger.
  The structural inference above still holds for repos that DO hand-write dynamic SQL, but don't assume every RuoYi-style repo has any — grep for `<if\|<foreach` first to check before treating this as a live concern.

## JS/TS + dynamic `import()` (field-verified 2026-07-16 on plus-ui / Vue Router)
- Blind: components registered via `() => import('@/views/x.vue')` (route-level code splitting — the standard Vue Router / React Router lazy-loading pattern) produce NO edge from the router file to the target component.
  Only statically-imported (`import X from '...'` at the top of the file) components get an `IMPORTS` edge.
- Confirmed on plus-ui's `src/router/index.ts`: it dynamically imports `login.vue`, `register.vue`, `error/404.vue`, `redirect/index.vue` and more — graph query for its outbound edges returns exactly ONE `IMPORTS` edge (the one static `import Layout from '@/layout/index.vue'`) and nothing else.
  Every dynamically-imported route component is invisible.
- Fallback: for "who/what routes to this Vue/React component", grep the component's file path (or its basename) directly inside router config files (`src/router/**`, `*.route.ts`, etc.) rather than trusting `cbm-trace.sh`/graph IMPORTS edges.
  Do not report a dynamically-lazy-loaded component as unused based on 0 inbound graph edges.

## PHP + Laravel
- Blind: Facades (`Cache::get` → container via __callStatic), Eloquent magic methods / dynamic scopes / relationship access, container string bindings (`app('foo')`, `bind/singleton` with string keys), Blade logic, event/listener wiring in providers.
- Fallback: For a facade, grep its accessor in `config/app.php` aliases or the Facade class `getFacadeAccessor()`, then trace the bound service in `app/Providers/*.php`.
  For Eloquent scopes grep `scopeXxx`.
  Read Blade templates directly.
- **2026-07-18/19 history**: an attempt to close this gap by indexing `pterodactyl/panel` drove host memory to within ~1GB of the machine's 64GB before a force-kill (see "`index_repository` memory usage" below); `hi.events` was deliberately not indexed with cbm either that pass, as a precaution. cbm's Laravel/PHP coverage stayed fully unverified through both passes.
- **Gap substantially closed 2026-07-20: `hi.events` (indexed as `hievents-bench`, 26,464 nodes/55,246 edges, no memory issue at all — see the memory-usage section below for why this repo's clean run doesn't contradict the `pterodactyl/panel` incident) was successfully indexed with cbm and probed directly.** Findings, all field-verified on this project's real code:
  - **Constructor-injected interface binding (standard Laravel DI) — cbm handles this WELL, and its recall is HIGHER than codegraph's on the one case both tools were tested against.**
    Clean single-owner case (`UserRepositoryInterface::findByIdAndAccountId`, not inherited by any other repository): `cbm-trace.sh` on the interface's qualified name at **depth 1** returned **12 direct callers**, an exact match with independent grep ground truth (12 distinct calling methods across 12 files) and with codegraph's own count on the identical symbol (also 12/12) — full agreement between both tools and grep here.
    Shared-base-method case (`RepositoryInterface::findById`, inherited un-overridden by ~15 different repository interfaces via `BaseRepository`): querying the CONCRETE impl node (`BaseRepository.findById`) returns **0 callers** — exactly the interface-attachment pattern already documented for Java, calls attach only to the interface declaration. Querying the INTERFACE node returned **92 direct callers** (depth 1) — every spot-checked class (`WebhookDispatchService`'s 5 methods, `EditProductHandler`'s 4 methods, plus 6 individually source-verified true positives among the callers codegraph misses, see below) was a genuine call site, and the count is consistent with a repo-wide grep of raw `->findById(` call expressions (~108 line-level call sites, collapsing to ~92-96 once multiple calls inside the same enclosing method are correctly counted as one `CALLS` edge rather than one per line).
    **This corrects a previously-published claim about codegraph on this exact symbol** (`codegraph-blindspots.md`'s "Laravel constructor-injected interface binding" section, and `SKILL.md`/`tool-collaboration-benchmark.md`'s citations of it): `cg-trace.sh` on the same interface node returns only **43** callers (reproduced exactly, unchanged from the original 2026-07-19 finding), and its own documented impl-node union adds only 7 more (43+7=50) — at most ~54% of cbm's 92. The original "43 real callers, cross-checked 100% against grep ground truth" claim was accurate only for the narrow subset it actually checked (the 16-file/18-call `eventRepository`-specific slice) — it was never cross-checked against the full ~15-interface shared-base population, and reads, uncorrected, as a much stronger completeness claim than the evidence supports. Six of cbm's callers absent from codegraph's list (`GetEventAction`, `MessagePendingReviewJob`, `OrderCancelService`, `IsAuthorizedService`, `ResendOrderConfirmationAction`, `SendWaitlistConfirmationEmailJob`) were individually confirmed via direct source read to be genuine `->findById(...)` call sites — codegraph's misses are real recall gaps, not cbm false positives. **Root cause pinned down same-day (2026-07-20 follow-up, see `codegraph-blindspots.md`'s corresponding section for the full evidence): it IS the fluent chain, and the earlier "ruled out" verdict above was wrong.** The original spot-check cited `EditProductHandler::validateChangeInProductType` as a counterexample ("a multi-hop chain that IS found") — a fresh direct source re-read shows this was a mistake: that method's only call site (`$this->productRepository->loadRelation(ProductPriceDomainObject::class)->findById(...)`) is in fact chained, and it IS one of the 42 methods codegraph misses, not an exception to the chain theory. A full, exhaustive (not sampled) re-classification of all 109 real `->findById(` call sites in the repo — DIRECT (`$this->prop->findById(...)` or `$var->findById(...)`, same line) vs CHAINED (`->findById(...)` as the tail of a preceding multi-line `->loadRelation(...)`/`->includeDeleted()` call) — found a 100% clean correlation with zero exceptions in either direction: all 50 methods codegraph finds have ≥1 DIRECT call site; all 42 methods it misses have ONLY chained call sites. This was also confirmed to be a genuine `codegraph` core extraction limitation, not a `cg-trace.sh` wrapper defect: calling the raw `codegraph callers` CLI directly at `-l 500`, bypassing the wrapper's bridging/dedup logic entirely, reproduces the identical 43/7 counts. Practical guidance: for a method inherited un-overridden across many different interfaces (or any method reached via a fluent builder chain), do not trust codegraph's single-node caller count as complete without a repo-wide grep cross-check for chained call sites specifically — and if both tools are installed, prefer cbm's number for this shape, since cbm's extractor does not share this chain-resolution gap (confirmed: all 42 chain-only callers were present in cbm's 92).
  - **Built-in Laravel Facades (`Cache::`, `Auth::`, `DB::`, etc. — target implementation lives in vendor code, not app code) are a CONFIRMED CLEAN BLIND SPOT for cbm.**
    `hievents-bench` uses `Cache::remember`/`Cache::get`/`Cache::has`/`Cache::put` at 6 real call sites across 4 files, with no `use Illuminate\Support\Facades\Cache` import at all (relying purely on Laravel's root-namespace facade alias, registered at framework boot, invisible to any static import-following). `cbm-find.sh` returns **zero nodes** for `Cache` as a class and zero for `remember` as a method (only unrelated `getRememberToken`/`setRememberToken` matched by substring) — there is nothing in the graph to even query, because vendor/framework code isn't indexed. This is a genuinely different case from the previously-tested APP-DEFINED Facade (`Activity::event()` on `pterodactyl/panel`, now deleted, where codegraph got 100% file recall) — that one worked because its accessor class lived in `app/` and got indexed like any other class. The blind spot is really "the facade's real target isn't part of the indexed source tree," not "facades in general" — codegraph was not re-tested against a built-in facade specifically and should not be assumed to fare any better here; the root blocker applies to any tool that only indexes application code.
  - **Eloquent relationship methods accessed via magic property (`$model->relationName`, no parens — Eloquent's `__get` intercepts and calls the declared relation method) are a CONFIRMED CLEAN MISS for cbm, same root cause and shape as the Eloquent-local-scope finding below.**
    Ground truth: `OrderRepository::getAttendees()` (`app/Repository/Eloquent/OrderRepository.php:129`) calls `$this->model->find($orderId)->attendees` — `Order::attendees(): HasMany` is declared at `app/Models/Order.php:29` (a second, unrelated `Event::attendees()` also exists at `app/Models/Event.php:39`, confirmed via cbm's own ambiguous-name disambiguation, which correctly listed both as separate candidates rather than guessing). `cbm-trace.sh` on `Order.attendees`'s exact qualified name returned **0 callers** — the property-read call site has no textual "call" shape (`attendees(` never appears) for the graph's `CALLS`-edge extraction to attach to, the same dynamic-magic root cause as the already-documented Eloquent local-scope miss.
  - **Eloquent local scopes (`scopeXxx` declared, invoked un-prefixed as `->xxx(...)`) remain untested for cbm specifically** — `hievents-bench` has zero `scopeXxx` methods anywhere in its codebase (confirmed via repo-wide grep), so this sub-case had no live example to test against on the one Laravel repo cbm could actually be indexed on. Only codegraph has field data here (confirmed 0/1 clean miss on `pterodactyl/panel`, now deleted) — do not assume cbm behaves identically until a repo with real local scopes is indexed with cbm.
  - **Methodology trap worth flagging for anyone re-running these probes: `cbm-trace.sh`'s 3rd positional arg is DEPTH (hop count), not LIMIT — unlike its sibling `cg-trace.sh`, whose 3rd positional arg IS a result limit.** Both scripts share the same-shaped usage line (`<name> [in|out|both] <N> [...]`), which makes this an easy trap: passing `200` as if raising a result cap (the natural instinct coming from `cg-trace.sh`) actually traces 200 HOPS deep, silently mixing in transitive/indirect callers with direct ones. Confirmed live: `findByIdAndAccountId` at depth 200 returned 15 "callers," including `ConfirmEmailAddressHandler.handle` — which does not call `findByIdAndAccountId` at all; it calls `EmailConfirmationService::confirmEmailAddress()`, which is itself a direct caller. At depth 1, the correct direct-callers-only count is 12, matching grep exactly. Always use depth 1 when comparing a `cbm-trace.sh` count against a hand-derived grep ground truth of DIRECT call sites; a higher depth answers a different question ("everything upstream within N hops") and will look like a recall gap (or, worse, an unexplained extra hit) if mistaken for the same measurement.

## Config-key binding: a genuine positive AND a newly-found asymmetry (field-verified 2026-07-20, RuoYi-Vue-Plus + plus-ui, reusing already-indexed repos)

- **Profile-specific YAML (`application-dev.yml`/`application-prod.yml`) — positive, previously "not confirmed to extend beyond the base `application.yml`."** `@Value("${snail-job.port}")` (`TestBroadcastJob.java:23`) is defined ONLY in the profile-specific files, never in the base `application.yml`. `cbm-grep.sh "snail-job"` (literal text search) correctly surfaces the key defined in BOTH `application-dev.yml:18` and `application-prod.yml:18` as `Variable` nodes. Confirmed: cbm's partial config-binding visibility extends to profile-specific YAML, not just the base file — matching codegraph's equivalent positive finding on the same symbol.
- **`.env` file coverage is NOT uniform across variants of the same env-file family — a new, previously undocumented asymmetry, worse than the existing "partial" framing suggested.** Tested `plus-ui`'s `VITE_APP_MONITOR_ADMIN` key, which is defined in BOTH `.env.development:15` and `.env.production:12` (confirmed both files are tracked in git, neither gitignored, `git status`/`git ls-files` checked directly). `cbm-grep.sh "VITE_APP_MONITOR_ADMIN"` returns exactly 3 matches: the real usage site (`monitor/admin/index.vue:8`), the type declaration (`env.d.ts`), and `.env.development` ONLY — **`.env.production` does not appear at all**, and `cbm-find.sh ".env.production"` confirms it has **zero nodes of any kind** in the index (not even a bare Module/File registration), while `.env.development` is fully indexed. Raising the grep limit (`-n 100`) does not change this — it is not a truncation artifact, `.env.production`'s content is genuinely absent from the index.
  Practical consequence: the existing guidance ("cbm-grep.sh partially models `.env` files") is not just incomplete per-key, it can be **completely blind to specific files within the SAME env-file family** depending on which variant cbm's file-discovery happens to pick up — a repo with multiple `.env.*` files must be checked file-by-file, seeing one variant indexed is no evidence the others are too.
- **`.properties` file ↔ `@Value` binding remains untested — no real example exists in the indexed repos.** Checked RuoYi-Vue-Plus's 8 real `.properties` files (`messages*.properties`, `spy.properties`): none are referenced via `@Value`/`@PropertySource` — the i18n bundles are consumed via Spring's `MessageSource` (a different binding mechanism entirely), and `spy.properties` is read directly by the p6spy driver, not Spring's externalized-config system. Every `@PropertySource`-style extra config file in this repo (`common-mybatis.yml`, `common-satoken.yml`, `generator.yml`) is YAML loaded through a custom `YmlPropertySourceFactory`, not a `.properties` file. This sub-case stays genuinely unverified for lack of a real test case, not for lack of trying.

## Go — first data point (`usememos/memos`, field-verified 2026-07-20) — confirms codegraph's finding, plus a naming-scheme nuance worth flagging

See `codegraph-blindspots.md`'s "Go" section for the full ground truth (a `Store` wrapper struct calling through a `Driver` interface field, itself held as a cross-package field on a business-logic struct — Go's single most ordinary DI pattern). cbm was tested against the identical symbol on the same index (7,396 nodes/26,513 edges, indexed clean with `--mode fast`, no memory issue on this ~803-file repo).

- `cbm-find.sh -l Method "CreateMemo"` (12 total results) shows entries for every concrete driver (`store.db.sqlite/mysql/postgres`), the interface (`store.Driver`), and unrelated proto-generated handlers — but **no entry at all for `Store.CreateMemo`** (`store/memo.go:109`, the wrapper method business code actually calls). Re-checked under `-l Function` instead: found `memos-bench.store.CreateMemo` (17 total results) pointing at `store/driver.go` — i.e. the INTERFACE declaration — with nothing distinct for the concrete `Store` wrapper's own declaration in `memo.go`. This suggests cbm's Go qualified-naming scheme is `<package-path>.<MethodName>`, WITHOUT the receiver/struct type name — unlike Java (`Class.method`) or PHP (`Class::method`) — so a same-package, same-method-name pair (the interface method `Driver.CreateMemo` and the concrete wrapper `Store.CreateMemo`, both literally in package `store`) may collide onto one queryable identity, or one is simply not extracted as a distinct node. **Not fully chased to a conclusion this pass** — flagging for a future Go-focused pass, since it changes what "the exact name" even means for cbm on Go the way it already matters for depth-vs-limit argument confusion documented above.
- Regardless of the naming-scheme nuance, the practical result is unambiguous: `cbm-trace.sh "memos-bench.store.CreateMemo" in 1 true` → **0 callers** — the one node this repo does expose under this name has no inbound edges at all, matching codegraph's 0/1 miss on the identical real caller (`server/router/api/v1/memo_service.go:115`). **Confirmed: cbm shares codegraph's severe Go cross-package struct-field-call blind spot.**

## Python + Django (field-verified 2026-07-20 on `healthchecks/healthchecks`, Round 9)

- **Dynamic URLconf composition (`include()` chains) — confirmed CLEAN blind spot, distinct from codegraph's positive result on the identical symbol.** `hc/urls.py`'s top-level `include("hc.front.urls")` → a second-level `include(check_urls)` → `hc/front/views.py:988`'s `details()` view (3 `include()` levels total). `cbm-trace.sh` on the exact `cbm-find.sh`-confirmed qualified name (`healthchecks-bench.hc.front.views.details`) returns a clean, empty `callers` result — route-registration is simply not represented as an edge in cbm's call graph at all. This is NOT the same shape as the shared MyBatis-XML/Vue-`globalProperties`-style blind spots elsewhere in this file: codegraph resolves this one correctly (see `codegraph-blindspots.md`'s Django section) via a `kind: "route"` edge, so treat Django URLconf as a **codegraph-only** capability, not a "native grep only" fallback. Full repro: `tool-collaboration-benchmark.md`'s "Round 9".
- **Custom Django Manager methods dispatched through `Model.objects` — a genuine POSITIVE result, and the clean counter-example this file had been missing for the PHP Eloquent-scope blind spot below.** `ProfileManager.for_user()` (`hc/accounts/models.py:62`, bound as `Profile.objects = ProfileManager()` at line 109) — defined and called under the identical name, no `scopeXxx`-style renaming convention involved. `cbm-trace.sh for_user in` at depth 3 found all 10 real direct callers across `hc/accounts/` and `hc/integrations/email/` (matching codegraph's direct-only count exactly) PLUS additional genuine hop-2/hop-3 transitive callers that codegraph's `in`-direction query doesn't surface at all — cbm's depth parameter is a real advantage here, not just parity. **The generalizable lesson: whether a framework-convenience method resolves is governed by whether the convention RENAMES the method between definition and call site, not by a blanket "ORM magic" label.** Laravel Eloquent's `scopeXxx()` → `->xxx()` renames and breaks name-based resolution (confirmed blind spot, see below); Django's Manager-method dispatch doesn't rename anything, so cbm's name-based matching still works cleanly. Don't extrapolate one ORM-convenience finding to another language/framework without checking for a rename first.
- **Decorator-application edges (`@decorator` → which functions it wraps) — confirmed CLEAN blind spot, shared with codegraph.** `hc/api/decorators.py`'s `authorize`/`authorize_read`/`cors`, real applications on multiple `hc/api/views.py` functions: cbm returns a clean, empty result for "which functions does this decorator wrap" (no false positives, but zero recall) — same shape as the existing framework-magic rows in this file (MyBatis XML, Vue/React dynamic `import()`). Native grep for `@decorator_name` above function definitions is the only reliable path. Folded into `SKILL.md`'s decision table alongside the other framework-magic constructs.

## Runtime bean-name lookup (Spring `getBean(dynamicName)`, field-verified 2026-07-16)
- Blind: a call resolved via a runtime-computed string (e.g. `SpringUtils.getBean(grantType + "AuthStrategy")`) has no static target at all — tree-sitter cannot know which of N `@Service("xxxAuthStrategy")` beans gets called, because the answer depends on a request parameter at runtime.
- Confirmed on RuoYi-Vue-Plus's `IAuthStrategy` strategy pattern (`ruoyi-admin/.../service/IAuthStrategy.java`): 5 implementations (`EmailAuthStrategy`, `PasswordAuthStrategy`, `SmsAuthStrategy`, `SocialAuthStrategy`, `XcxAuthStrategy`), each registered under a computed bean name, looked up via `SpringUtils.getBean(beanName)` inside a `static` interface method.
  No amount of graph querying will connect `AuthController` to any specific implementation — this is not a graph gap, it is genuinely undecidable without running the code.
- Fallback: grep the bean-name construction pattern (here `+ IAuthStrategy.BASE_NAME` / `@Service("..." + ...)`) to enumerate all candidate implementations by hand; do not expect `cbm-trace.sh` to find a path from caller to any specific impl.

## Cross-repository analysis (front-end + back-end split, e.g. this monorepo pair)
- Each repo is indexed as its own independent `project` (verified: `plus-ui` and `RuoYi-Vue-Plus` are separate graphs, separate node/edge counts, no shared nodes).
  There is currently no multi-root/aggregated-workspace graph — this is a deliberate, working setup, not a missing feature blocking anything documented here.
- Consequence: a question spanning both repos (e.g. "which frontend page calls this backend API") requires TWO separate graph calls (one per project, via `--name`/`-p` project targeting) plus manual correlation of the Vue API-call string against the backend `Route` node path.
  The graph will not do this join for you.
- This limitation is identical for codegraph — its index is likewise scoped to whatever directory it was run in (no `--name` needed, but the same one-index-per-repo boundary), confirmed by direct test, not just inferred from the architecture.
- **Field-verified recipe (2026-07-18, see `references/tool-collaboration-benchmark.md` for the full run)**: picked 3 real frontend API request URLs from `plus-ui/src/api/system/dict/*.ts`, grepped the URL path fragment directly against `RuoYi-Vue-Plus`'s Java controllers (`grep -rn "<path-fragment>" --include=*.java`), and independently cross-checked with `cg-find.sh -k route` on the backend side — 3/3 matched a real endpoint.
  This is the only reliable path for this question shape today; treat any future "cross-repo query" flag on either CLI's changelog as worth re-testing, but do not assume one exists.

## `cbm-cypher.sh`'s officially-relied-on aggregate templates — 2 of 5 were silently broken, now fixed, plus 3 more issues found in a same-day code-review pass (field-verified 2026-07-17)

The decision table has always pointed whole-graph questions (dead code / hubs / cross-layer / routes) at `cbm-cypher.sh` as this skill's flagship advantage over per-symbol tools — that claim had never actually been executed and checked end-to-end before this pass.
Turned out 2 of the 5 templates were broken in ways that produced a confidently-formatted, plausible-looking WRONG answer rather than an honest error — the same dangerous failure shape as `codegraph explore` on aggregate questions, see the sibling `codegraph-blindspots.md` in this same skill.

- **`hubs` was completely non-functional.**
  The shipped query was `MATCH (c:Class) RETURN ... ORDER BY c.degree DESC LIMIT 20`.
  Ran `MATCH (c:Class) RETURN keys(c) LIMIT 3` directly: the schema is `name, qualified_name, label, file_path, start_line, end_line` — **`degree` does not exist as a property on Class nodes at all.**
  `ORDER BY` on an always-null column is a silent no-op, so the "top 20 hubs" was really just native scan order.
  Confirmed the practical damage on RuoYi-Vue-Plus: `TestTree` (a test-only domain class) and `BackProcessBo`/`SysDeptVo` (plain data objects) ranked in the top 20 "god classes," while zero real utility/base classes appeared.
  Fixed in `cbm-cypher.sh` by aggregating real inbound `CALLS` edges across each class's methods (`MATCH (m:Method)<-[:CALLS]-() WHERE m.parent_class IS NOT NULL RETURN m.parent_class, count(*) ... ORDER BY count(*) DESC`) rather than counting direct calls to the Class node itself.
  Tried that naive version first and it undercounts badly, because a `CALLS` edge into a `Class` node is a **constructor call only** (`new Foo()`); ordinary `obj.method()` calls attach to the `Method` node, not the class.
  The fixed query's top results — `StringUtils` (279), `R` (259), `LoginHelper` (230), `BaseMapperPlus` (133), `StreamUtils` (110), `BaseController` (70) — are exactly the utility/base classes a Java engineer would expect to top this list, a dramatic accuracy jump from the broken version.
  Caveat confirmed by testing on plus-ui (the JS/TS/Vue repo): this class-level aggregation returns 0 rows there, because plus-ui has essentially no `Method`-labeled nodes (1 total) — Vue/TS code in this graph is modeled as `Function`, not class methods.
  This is not a bug, it's a real modeling mismatch: **the fixed `hubs` template only works for class/OOP-heavy repos (Java, etc.); it has no signal for a function-oriented JS/TS/Vue codebase**, which would need a `Function`-level equivalent (not built — out of scope for this pass, note it if asked to extend this template).
- **`cross-layer` hard-failed on its own documented default invocation.**
  Running `cbm-cypher.sh cross-layer` with zero args (the way the SKILL.md decision table shows it) returned `unexpected operator at pos 38` on every single call — not a graceful `hint`, a raw Cypher parser error surfaced straight to the caller.
  Isolated the cause by testing 3 query variants directly: `coalesce(a.file_path, a.file) CONTAINS '...'` in a `WHERE` clause breaks the parser, while the same `coalesce(...)` works fine inside `RETURN`, and a plain `a.file_path CONTAINS '...'` (no coalesce) works fine in `WHERE`.
  Also confirmed via `WHERE n.file_path IS NULL` that 0 of 2163 Method nodes in this repo actually lack `file_path`, so coalesce's `.file` fallback was never doing anything useful here anyway.
  Fixed by dropping `coalesce()` from the `WHERE` clause only (kept in `RETURN` for display).
  The fixed query returns exactly 4 real controller→mapper layer violations on RuoYi-Vue-Plus (e.g. `TestBatchController.add` calling `insertBatch` directly on a Mapper) — a small, plausible number for a codebase that mostly routes through its service layer correctly, which is itself a useful signal (the rare violations are worth looking at precisely because they're rare).
- **`dead-code` (the plain `Function`-label template, distinct from `dead-code-methods`) has its own systematic false-positive pattern, not previously documented.**
  Every Java interface method declaration turns out to be double-registered in the graph as BOTH a `Method` node AND a separate `Function` node at the identical file/line — confirmed via `MATCH (n) WHERE n.name = "..." RETURN labels(n), n.file_path` on 5 different interface methods, each producing exactly this Method+Function pair on the interface file, plus a third `Method` node on the impl side.
  Real `CALLS` edges from real callers attach only to the `Method`-labeled twin (consistent with the interface/impl section above) — the `Function`-labeled twin is structurally incapable of ever having an inbound edge, so `dead-code` reports **every single interface method in the codebase as dead**, independent of and in addition to the already-documented impl-side blind spot (which only affects `dead-code-methods`, the `Method`-label template).
  This is NOT a blanket "Function label is broken" finding — confirmed 95 of 449 Function nodes in this repo DO have real inbound CALLS edges, and a genuinely-unused enum method (`FormatsType.getFormatsType`) was correctly flagged with zero grep hits repo-wide to confirm it really is dead.
  The false-positive is specific to interface method declarations; `cbm-cypher.sh` now emits a warning on every `dead-code` call telling the caller to cross-check any `*Service`/`I*`-shaped hit with `cbm-trace.sh` first.
- **`dead-code-methods` has a further false-positive shared with codegraph's own call-chain commands, not fixable by cross-checking with the other tool.**
  A candidate flagged dead by `dead-code-methods` (`SysDictTypeServiceImpl.selectDictTypeByType`, an `@Cacheable` method) turned out to have a real caller reached only through `SpringUtils.getAopProxy(this).selectDictTypeByType(...)` — a self-invocation through the Spring AOP proxy, done specifically to route the call through the proxy so `@Cacheable` actually triggers.
  `cg-trace.sh`/`cg-node.sh` were run as an independent cross-check and ALSO missed this call site — see `codegraph-blindspots.md`'s dedicated section for the full repro on that side.
  The two signals agreeing here is not corroboration: both share the same underlying gap (neither tool's static extractor resolves a call reached through `getAopProxy(this)`), so before trusting any dead-code verdict on an AOP-annotated method (`@Cacheable`/`@CachePut`/`@CacheEvict`), grep `SpringUtils.getAopProxy(this).<method>(` as a standing check, the same discipline as the existing `SpringUtils.getBean(X.class)` spot-check.
- **`routes` was already accurate** — spot-checked its output against real controllers (`/monitor/logininfor/list`, `/demo/sms/sendAliyun`, etc.), all well-formed and traceable to real `@RequestMapping`/`@GetMapping` sites.
  No change needed.
  `Class` nodes were separately found to ALSO represent MyBatis XML `<select>/<update>` statement elements — e.g. a Class node literally named `select` with `qualified_name` ending `TestDemoMapper.select`, `file_path` pointing at the `.xml`.
  This doesn't affect `routes` (different node label) but is why the naive "count calls into Class nodes" hub query above needed the parent_class-aggregation rewrite rather than a simple `.xml`-path filter; noted here in case a future template touches raw `Class` nodes again.
- **Methodology note**: all five templates were executed directly against RuoYi-Vue-Plus's real index this pass (not read from the script source and assumed correct) — the same discipline as the interface/impl and dynamic-`import()` findings above.
  Re-run this spot-check after any `codebase-memory-mcp` version bump; a schema change could silently reintroduce or shift these issues.
- **Same-day code-review follow-up found 3 more issues in this same template set, all now fixed.**
  A structured review of the just-fixed script (SOLID/security/quality pass, not a fresh field test of new query shapes) surfaced these by re-reading the fix with the discipline of "would a wrong answer here look plausible" rather than just "does it run":
  1. **Silent truncation, present on every fixed-LIMIT template, never previously checked.**
     None of the templates compared their returned row count against their own `LIMIT`, so a true result set bigger than the cap was returned looking exactly like a complete one — the same dangerous shape as the `hubs`/`cross-layer` bugs above, just not yet caught.
     Ran a direct `count(*)` against each template's `WHERE` clause with no `LIMIT` and compared: `routes` LIMIT 200 vs true count **303** (103 hidden, 34%); `dead-code` LIMIT 100 vs true count **348** (248 hidden, 71%); `dead-code-methods` LIMIT 100 vs true count **1159** (1059 hidden, 91%).
     This directly contradicts the "`routes` was already accurate — no change needed" line earlier in this section — that check only verified individual row correctness (real controllers, well-formed paths), never verified the total count against the cap, so a third of the real routes were being silently dropped the whole time.
     Fixed generically: `cbm-cypher.sh` now runs a cheap follow-up `count(*)` whenever a template's result count exactly equals its `LIMIT`, and emits a stderr warning with the real total and how many rows are hidden.
     `hubs` is deliberately exempt — a "top 20" ranking has no "true total" to compare against, capping there is the intended behavior, not data loss.
  2. **`cross-layer`'s `layerA`/`layerB` arguments were spliced into the Cypher string with no escaping.**
     Confirmed live: passing an argument containing a single quote (`/controller/' OR '1'='1`) crashes the parser (`expected token type 85, got 86`) instead of being treated as a literal path-fragment filter — an injection-shaped input-handling defect, not merely a crash-on-weird-input bug, even though a full working injection payload for this specific restricted Cypher grammar was not constructed.
     Fixed by stripping `'` and `\` from both arguments before interpolation; re-tested with the same payload above post-fix — returns a normal empty-result JSON instead of crashing.
  3. **This script's underlying `cbm_call` (in `scripts/_project.sh`) has no JSON-validation safety net, unlike this skill's codegraph-side `cg_call()` (see `scripts/_gate.sh`).**
     Before today's fixes, both the `hubs` degree-ordering bug and the `cross-layer` coalesce bug manifested as a *raw, unhandled crash* reaching the caller — not a graceful `{"error", "hint"}` response — because nothing between the Cypher engine and stdout ever validated the output was JSON.
     This violates the "every script returns valid JSON with a hint, never a raw crash" contract both `SKILL.md`s describe as the mandatory-sequence guarantee.
     Fixed LOCALLY inside `cbm-cypher.sh` (a `run_query` wrapper, same tempfile + `jq empty` pattern as `cg_call()`) so this script now upholds that guarantee regardless of future template bugs.
     Not fixed: the shared `_project.sh::cbm_call` itself, which `cbm-find.sh`/`cbm-grep.sh`/`cbm-trace.sh`/`cbm-impact.sh`/`cbm-snippet.sh`/`cbm-arch.sh` all call directly — those 6 scripts still have no equivalent protection.
     Widening the fix to the shared helper would touch every script in this skill in one pass; deliberately left as a follow-up rather than done inside an already-large review, to keep this change reviewable and its regression surface small.

## `cbm-find.sh -s` (semantic business-term search) — 2 real bugs found and fixed, plus a language limitation that is NOT fixable (field-verified 2026-07-17)

This skill's own decision table pointed Chinese business-language questions (业务词) at `cbm-find.sh -s` as the dedicated tool — that path had never been executed and checked end-to-end before this pass, the same gap that produced the `cbm-cypher.sh` bugs above.

- **Bug 1 (fixed): `semantic_query` was sent as a JSON string, not an array.**
  The CLI's own `--help` for `search_graph` is explicit: `--semantic-query <array> ... MUST be an ARRAY of keyword strings ... NOT a single string`. The script was building `{semantic_query:$q}` with `$q` a plain string (`jq --arg`).
  This does NOT error — it silently returns near-random results with no signal that anything is wrong: on this repo, the ENGLISH string `"dict label"` scored top hit ~0.03 with irrelevant results (`AuthTopIamRequest`, `selectDeptLeaderById`, `doFilter` — nothing to do with dictionaries), while the equivalent 2-element array `["dict","label"]` scored ~0.97 on exactly the right hits (`getDictValue`, `getDictLabel`, `getAllDictByDictType`, all in `DictService`/`SysDictTypeServiceImpl`) — same query text, same repo, same index.
  Fixed: the script now splits the query on whitespace into a JSON array before calling (`jq -R '[splits("[ \t]+")] | map(select(length>0))'`), matching the CLI's own documented example shape (`["send","pubsub","publish"]`).
- **Bug 2 (fixed): the script's output formatter read `.results`, but semantic hits live in a separate `.semantic_results` field.**
  Confirmed directly: even after fixing bug 1, `.results` for a `semantic_query` call is an unranked dump of arbitrary graph nodes (e.g. `.codebase-memory/.gitattributes`, `.gitee/ISSUE_TEMPLATE/*.yml` — literally repo config files, nothing to do with the query) — the real ranked hits with `score` were sitting untouched in `.semantic_results` the whole time. Fixed: the script now reads `.semantic_results` when `-s` is set.
- **Limitation (NOT fixable in this script): the embedding model does not support Chinese query text.**
  Even with both bugs fixed, a Chinese business term scores in the same near-random range as the string bug, regardless of whether it's sent as a single-element array (the whole phrase, e.g. `["字典标签"]`) or split into multiple single-character/word elements (`["退款","审核"]`) — top scores observed were 0.02–0.10 with irrelevant hits (rate-limiter config, WebSocket handlers, monitor admin `main()` — no relation to the query), essentially indistinguishable from noise. This reproduces on every Chinese query tried, not an isolated phrase.
  This is a property of the underlying vector model, not something a wrapper script can work around.
  **Correct fallback for Chinese business terms: `cbm-grep.sh` (`search_code`, literal text match), not `cbm-find.sh -s`.** RuoYi-Vue-Plus's Javadoc comments are written in Chinese and contain the business term verbatim (e.g. `/** 根据字典类型和字典值获取字典标签 */` on `DictService.getDictLabel`) — confirmed `cbm-grep.sh '字典标签'` returns 14 correctly-ranked, genuinely relevant results (`DictService`, `SysDictData`, `SysDictDataBo`, the `dictLabel` field, etc.) in 398ms, because it's a literal grep-style match over source text, not an embedding lookup. BM25 full-text search (`search_graph`'s `query` param, NOT currently used by any script in this skill) was also tested and also fails on Chinese — it tokenizes on whitespace/camelCase boundaries per its own `--help`, and Chinese has neither, so a Chinese query degrades to the same unranked full-graph dump. Only the literal-match path (`search_code`) is language-agnostic here.
  `cbm-find.sh` now emits this same guidance as a `hint` whenever a semantic query's top score is below 0.3 (a threshold picked with wide margin: confirmed-good matches score ~0.9+, confirmed-broken ones score ~0.02–0.10).
- **Methodology note**: root-caused by isolating each variable independently rather than accepting the first plausible explanation — tested string-vs-array holding language constant (both English: 0.03 vs 0.97, confirming Bug 1 is real and language-independent) AND array-vs-string holding language constant (both Chinese: ~0.02–0.03 either way, confirming the Chinese limitation survives the Bug-1 fix and is a separate root cause, not the same bug restated).

## CALLS-edge resolution is name-only — same-named methods on UNRELATED classes collide, not just interface/impl pairs (field-verified 2026-07-17)

Distinct from the Java interface→impl blind spot above (that one is about which NODE of a *related* interface/impl pair gets the edge). This one is about the edge landing on the wrong *method entirely*, on a class with no relationship to the real target.

- Traced all inbound `CALLS` edges to `DictService.getDictLabel(String dictType, String dictValue, String separator)` (a 3-arg business method) and verified each one against source by hand. **6 of 10 edges (60%) were false positives**: they were really calls to `getDictLabel()` — a zero-arg Lombok-generated getter on `SysDictDataVo`/`SysDictData` (from a `dictLabel` field under `@Data`), a completely unrelated class with no inheritance/interface relationship to `DictService`. Examples: `SysDictTypeServiceImpl.getAllDictByDictType` line `map.put(vo.getDictValue(), vo.getDictLabel())` — `vo` is a `SysDictDataVo`, so `.getDictLabel()` is the Lombok getter, not the business method; a method reference `SysDictDataVo::getDictLabel` inside `StreamUtils.toMap(...)` produced the same false attachment.
  Only 4 of 10 were real (verified via source: `ExcelDictConvert.convertToExcelData`, `DictPatternValidator.isValid`, `DictTypeTranslationImpl.translation`, `SysNoticeController.add` — all genuinely call `SpringUtils.getBean(DictService.class).getDictLabel(...)` or `dictService.getDictLabel(...)`).
- Root cause: `CALLS` edges resolve by simple method name only — no parameter count, parameter type, or receiver type check. Any business method whose name collides with a same-named getter/setter (the single most common Java naming collision, since `@Data`/`@Getter` on every DTO/VO/BO/Entity generates `getX()`/`setX()`/`isX()` for every field) will have its caller list polluted by every unrelated call to that getter anywhere in the codebase.
  This is NOT rare in a Lombok-heavy codebase — this repo uses `@Data` on essentially every domain/DTO/VO/BO class, so any business method named `getDictLabel`/`getDictValue`/`getName`/`getStatus`/etc. is at risk purely from naming coincidence.
- Consequence for `cbm-trace.sh`: a caller/callee list for a method whose name starts with `get`/`set`/`is` (or otherwise matches a plausible field-accessor shape) cannot be trusted at face value — always spot-check a sample of the returned callers against source (does the receiver's declared type actually have this method as the business method, not a generated accessor?) before reporting a caller count or "who calls X" answer as fact.
  A quick heuristic: business methods on Service/Mapper interfaces almost always take ≥1 parameter (queries need at least a key; commands need a payload) while Lombok getters take 0 — a caller whose call-site has zero arguments is a strong hint it's actually hitting the accessor, not the business method.
- Consequence for `cbm-cypher.sh dead-code`/`dead-code-methods`: the same name-only resolution means a method can look "used" (nonzero inbound `CALLS`) purely because an unrelated getter with the same name is called elsewhere — a false NEGATIVE for dead-code detection, the opposite direction from the interface/impl false-positive already documented above. Both directions of error stem from the same root cause (name-only resolution) and can coexist on the same method.
- Corroborating positive data point: this is a precision problem on ambiguous names, not a blanket failure. Ran the same source-verification against 8 zero-inbound-degree candidates that are NOT interface methods and NOT getter/setter-shaped (`RedisUtils.getCacheMapKeySet`/`.getMultiCacheMapValue`/`.addMapListener`, `StringUtils.containsAnyIgnoreCase`, `SpringUtils.getAliases`, `DateUtils.difference`, `EncryptUtils.encryptBySm3`, `MybatisExceptionHandler.findCause`) — all 8 confirmed genuinely dead via repo-wide grep (zero external references). Concrete utility-class methods with distinctive names are reliable; the risk is specifically name-collision-prone methods (getter/setter shape, or a name repeated across unrelated interfaces like `updateById`/`selectList` inherited from MyBatis-Plus's `BaseMapper`).

## `index_repository` project-identity resolution is sensitive to how `--repo-path` is given (field-verified 2026-07-17, defensive note — no script in this skill triggers it)

Running `codebase-memory-mcp cli index_repository --repo-path . --mode fast` (relative path, no `--name`) from within `D:/data/RuoYi-Vue-Plus` did NOT update the existing `RuoYi-Vue-Plus` project — it silently created a SECOND, separate project (observed name: `D-data-RuoYi-Vue-Plus`, derived from the resolved absolute path) pointing at the same `root_path`. ## `index_repository` memory usage — a serious, cross-platform operational risk, not a query-accuracy gap (field-verified 2026-07-18)

- Attempted to index `pterodactyl/panel` (1,289 files, PHP 64.8% / TS 20.9% / Blade 11.7%, shallow-cloned) with `codebase-memory-mcp cli index_repository --mode fast` on a 64GB-RAM Windows machine, specifically to close the Laravel+PHP/React+TS evidence gap in this file.
  First attempt: the indexing worker crashed (`outcome=exit_nonzero`), contained by the supervisor process, no partial index written. Retried per the tool's own hint ("Re-run to retry").
  Second attempt: no crash, but system free memory dropped from ~58GB to ~6GB while the process was still mid-indexing (confirmed via `Get-Process`/`Get-CimInstance Win32_OperatingSystem`), and was still falling when the process was force-killed as a safety measure after roughly 30 minutes of runtime with no completion in sight. Free memory returned to ~58GB immediately after the kill, confirming this process (not something else) held the memory.
  The tool's own startup log line (`level=info msg=mem.init budget_mb=32587 total_ram_mb=65174`) shows it initializes with an intended budget of about half the host's RAM — actual usage in this run exceeded even that stated budget before being killed.
- **This is a known, actively-tracked upstream issue class, not specific to this repo, this OS, or this machine.** Confirmed via the project's own GitHub issue tracker (`DeusData/codebase-memory-mcp`), not assumed:
  - Windows: [#581](https://github.com/DeusData/codebase-memory-mcp/issues/581) (process grows to 50+GB virtual memory over hours, crashes Windows), [#832](https://github.com/DeusData/codebase-memory-mcp/issues/832) (memory not released after indexing, 20GB+ RSS growing ~600MB/min), [#775](https://github.com/DeusData/codebase-memory-mcp/issues/775) (indexer deadlock/RAM blowup reaching ~62GB on a 64GB machine, "crashed the whole PC"), [#1084](https://github.com/DeusData/codebase-memory-mcp/issues/1084) (high CPU/memory on 24GB Windows machines).
  - macOS: [#580](https://github.com/DeusData/codebase-memory-mcp/issues/580) (feature request for a `--max-memory` flag — STILL OPEN, not shipped — reporter's 8GB M1 Air hit 8-11GB RSS indexing a ~6,000-file repo), [#765](https://github.com/DeusData/codebase-memory-mcp/issues/765) (PHP trait collision triggering unbounded recursion, 18-20GB RSS on a 36GB Mac, SIGKILL/exit 137), [#317](https://github.com/DeusData/codebase-memory-mcp/issues/317) (dump-phase crash on a large TS monorepo, arm64, peak RSS ~5-5.5GB).
  - Linux: [#363](https://github.com/DeusData/codebase-memory-mcp/issues/363) (does not respect cgroup CPU/memory limits inside a container), [#1070](https://github.com/DeusData/codebase-memory-mcp/issues/1070) (indexer killed by signal 9 on an 8GB arm64 Linux machine even with an explicit small `budget_mb` configured — the budget setting itself did not prevent the OOM kill).
  - Umbrella tracking issue: [#593](https://github.com/DeusData/codebase-memory-mcp/issues/593).
  This spans all three major OSes and ranges from "uncomfortably high" (5-20GB on constrained hardware) to "crashes the whole machine" (50-62GB on well-provisioned Windows machines) — treat it as a live, unresolved risk class for THIS binary, not a one-off fluke of this session's environment.
- **Practical guidance for anyone running `index_repository` on a repo this skill hasn't already indexed successfully**: watch host memory during the first index of any new, moderately-large (roughly &gt;1,000 file) or mixed-language repo, especially on Windows; be ready to kill the process if usage climbs unbounded rather than waiting it out; there is currently no supported flag to cap memory (the `--max-memory` request in #580 is still open) — `--mode fast` reduces analysis scope but did NOT prevent the blowup observed here.
  Contrast with `codegraph`, which indexed the identical 1,289-file repo in 2.6 seconds with no memory concern observed — this is a real, practical asymmetry between the two tools worth factoring into which one to install first on a large or unfamiliar repo, independent of either tool's query-accuracy tradeoffs documented elsewhere in this file.
- **Round 8 update (fresh `pterodactyl-panel-bench` re-clone, ~1,276 source files — this round ran the controlled full-vs-fast mode comparison the earlier attempts never did). Three new operational facts, each changing what "be ready to kill it" should mean in practice:**
  1. **Switching `--mode` is NOT a mitigation.** Default (full) mode with `--persistence true`: the indexing worker's memory hit **37.8GB within ~2 seconds** of starting and **52.4GB by ~10 seconds** (65GB total RAM machine; the tool's own logged `budget_mb=32587` was blown through almost immediately). Retried with `--mode fast` — whose documentation says it skips the similarity/semantic-edge computation: **13GB by 10 seconds, 39.4GB by 20 seconds**, a comparable growth rate, killed the same way. Since the blowup survives removing exactly what `fast` removes, the memory black hole most plausibly lives in the type-aware LSP call/usage-resolution step that every mode runs — an inference from one repo's controlled comparison, not a proven root cause, but enough to stop treating "try a lighter mode" as a fallback plan. Note this run detonated in seconds where Round 2's `--mode fast` attempt climbed for ~30 minutes before the kill — the failure onset varies run to run even on the same repo; a calm first minute is not evidence of a safe run.
  2. **Killing the worker process alone does not stop it.** The CLI runs a supervisor + worker pair, and force-killing only the worker made the supervisor immediately spawn a fresh worker that began climbing again. Both processes have to be killed to actually end an indexing attempt — worth knowing before the machine is already swapping.
  3. **Both aborted attempts failed clean**: `list_projects` showed no partial/corrupt project entry afterward, and the repo tree had no leftover persistence artifacts — consistent with Rounds 2/5. The damage mode is host-memory exhaustion during the run, not index corruption after it.
  Defensive note on this skill's own scripts: `_project.sh`'s "repo not indexed" error hint helpfully includes a ready-to-paste `index_repository --repo-path ... --name ...` command — on a repo shaped like this one, pasting that command without watching memory reproduces this incident. The hint is correct for the common case; it just cannot know which repos are detonators, so the human (or agent) following it has to.
  On predicting WHICH repos are risky: the only confirmed detonator so far is `pterodactyl/panel` (~1,276 source files, PHP+TS+Blade), reproduced on every attempt across three rounds, while `hievents-bench` (~2,200 files, also PHP+TS), `memos-bench` (~803 files, Go), and `spatie/laravel-permission` (106 files, PHP) all indexed clean — so ">1,000 source files + mixed non-JVM languages" raises the prior of hitting this known upstream issue class (which includes a PHP-specific unbounded-recursion trigger, #765) but does not cleanly predict it, and no precise boundary can be drawn from one detonating repo. The economical default on a repo matching that rough shape: index with codegraph only (seconds, covers most question shapes) and skip cbm indexing unless a cbm-exclusive capability (dead code / hubs / cross-layer) is actually needed badly enough to justify a monitored attempt.
- **Round 10 — the ">1,000 files + mixed language" heuristic above is WRONG, and the real trigger has been isolated down to a single file (2026-07-20), disproven and replaced by a minimal, deterministic repro.** Prompted directly by the user's hypothesis that PHP+TS mixing was the cause; `hievents-bench` (same language mix, larger file count, indexed clean three times) already contradicted that before this round started, so the investigation bisected `pterodactyl-panel-bench` itself rather than comparing across repos. Method: built an automated PowerShell watchdog (poll every 250ms, kill both the worker and any respawned worker the instant total working set exceeds 6GB or free memory drops >15GB from baseline — necessary because the full-repo blowup previously hit 37.8GB within 2 seconds, too fast for manual reaction) and ran `index_repository --mode fast` against progressively smaller file subsets copied into disposable throwaway git repos:
  - `resources/scripts/` alone (287 TS/TSX files, React frontend) — **clean, instant** (1,286 nodes/2,832 edges). `app/Http/` alone (222 PHP files — controllers, requests, middleware) — **clean, instant** (1,263 nodes/2,829 edges). Both rule out "PHP", "TS", and "large file count in general" as the trigger.
  - `app/Models/` alone (38 files) reproduced the full blowup curve. Bisecting `Models/` by half (16 files, deliberately excluding `Server`/`User`/`Node`, the 3 most relationally-complex models) still reproduced it — ruling out any single "hub" model. Narrowing further: **`app/Models/Database.php` + the shared `Model.php` base class — just 2 files — is sufficient to reproduce the entire multi-round incident on its own**, at the same growth rate as the full 566-file backend (`app/Server.php` + `Model.php`, by contrast, is clean despite Server being the single most relationally-complex model in the app, 15 relation methods, tested first as the obvious suspect and ruled out).
  - **Isolated the exact trigger inside `Database.php` with two confirming/disconfirming edits, both verified live**: (a) deleting only the file's two relation methods (`host()`/`server()`, both `BelongsTo` with a `@return \Illuminate\Database\Eloquent\Relations\BelongsTo<TargetClass, $this>` PHPDoc generic, `$this` used as a literal generic type argument referring to the enclosing class) while keeping the class name `Database` and everything else byte-identical — **fixed it, clean index**. (b) Reverting that and instead renaming ONLY the class declaration (`class Database extends Model` → `class DatabaseXYZ extends Model`), keeping both relation methods and their generic annotations 100% byte-identical — **also fixed it, clean index**. Neither condition alone in the other's absence explains it (`Server.php` has the identical `BelongsTo<X, $this>` annotation pattern on 14 different relation methods and is clean; a same-file-count swap test replacing `Database.php` with `DatabaseHost.php` in an otherwise-identical 6-file set stayed clean, ruling out a pure file-count threshold and ruling out "any model with this generic-annotation shape").
  - **Conclusion: this is a name-collision bug specific to a user class literally named `Database` combined with a self-referential generic relation annotation (`<Target, $this>`) on at least one of its methods — not a repo-size, file-count, or language-mix property at all.** The most plausible mechanism (not verified against the tool's closed source, which this skill has no access to): `codebase-memory-mcp`'s own persistence/storage layer is internally built around the vocabulary "database" (its artifact is literally `graph.db.zst`, its CLI is `codebase-memory-mcp`) and some internal lookup or cache keyed by bare class/symbol name collides when a user-indexed class is *also* named `Database`, and the collision compounds specifically when that class's own methods contain a self-type-referencing generic (`$this`) the resolver has to expand recursively against the (now-colliding) symbol table entry. This reframes the standing ">1,000 files / mixed language" guidance above from a real predictor into a coincidental correlate — `pterodactyl/panel` happens to be large AND mixed-language AND has a model named exactly `Database`, and only the third property actually matters. Two sibling risk names spotted in the same repo during this investigation but NOT individually tested: `app/Models/Session.php`, `app/Models/Model.php`'s own base-class role — worth a follow-up bisection before generalizing this into a blanket "avoid framework-reserved-sounding class names" rule.
  - **Practical guidance, replacing the old size/language heuristic**: before running `index_repository` on an unfamiliar repo, `grep -rn "^class \(Database\|Cache\|Config\|Session\|Container\|Storage\|Log\)\b"` for a same-named class combined with a self-referential generic (`, \$this>` or `<static>`) return-type annotation on any of its methods is a much sharper, cheap, zero-risk pre-check than eyeballing file count or language mix — that combination, not repo size, is what's actually been confirmed to detonate this tool. All throwaway bisection fixtures (`ptero-bisect-*` directories under `D:/data/`) were deleted after this investigation; only the two confirming/disconfirming file edits above are needed to reproduce the finding from a fresh `pterodactyl-panel-bench` clone.

## `cbm-*.sh` wrapper-script defects found by cross-checking every script against its underlying CLI's own `--help` (field-verified 2026-07-20 on RuoYi-Vue-Plus/plus-ui)

The `cg-trace.sh` silent-20-cap bug (see `codegraph-blindspots.md` and `tool-collaboration-benchmark.md`'s Round 3 section) was found by accident while probing a new repo, not by systematically checking this skill's own wrapper scripts against the CLIs they call. This pass did that systematic check on the `cbm-*` side: every `cbm-*.sh` script's assumptions (hardcoded params, output field names) were diffed against a fresh `codebase-memory-mcp cli <tool> --help` dump and, where practical, against a live call on the still-indexed `RuoYi-Vue-Plus`/`plus-ui` projects. It surfaced four more real defects, all now fixed.

- **`cbm-find.sh` hardcoded `limit:20` — LOWER than `search_graph`'s own documented default of 200 — and discarded the `total`/`has_more` fields the CLI ships specifically to detect truncation.**
  `search_graph --help` is explicit: `--limit <integer> Max results per call. Default 200. Response carries 'total' (full match count) and 'has_more' (true if truncated) so callers can detect the limit and paginate.` The script ignored all of that: it always sent `limit:20`, and its output-formatting `jq` step rebuilt a brand-new object (`{count, results, hint}`) that never referenced `.total`/`.has_more` at all.
  Confirmed live: `cbm-find.sh -l Route '.*'` (label-filtered, matching every Route node) returned exactly `count: 20` with no warning, while a direct `search_graph` call with `label:"Route"` on the same project reports `total: 303, has_more: true` — 283 real routes silently invisible, the identical failure shape as the `cg-trace.sh` bug, just on the `search_graph` side instead of `callers`/`callees`.
  This directly explains why the Decision table's "exhaustive listing of a kind" row had to say "exhaustiveness NOT field-verified for non-route kinds, say so" about `cbm-find.sh -l <Label>` — the tool wasn't merely unverified, it was hard-capped at a number lower than even its own CLI's default, with the one signal designed to catch this thrown away.
  Fixed: default limit raised to 200 (matching the CLI's own default), a new `-n LIMIT` flag added to override it, and `total`/`hasMore` now flow through into both the JSON output and the `hint` field (mirroring the `possiblyTruncated` pattern `cg-trace.sh` already uses).
- **`cbm-trace.sh`'s own "0 results" hint checked field names that don't exist in `trace_path`'s real response — it reported "0 results" on EVERY call, real results or not.**
  `trace_path`'s actual output keys, confirmed live, are `callers` (inbound direction), `callees` (outbound), or both (direction `both`) — never `paths` or `results`. The script's hint logic tested `(.paths // .results // [])|length == 0`, which is always true regardless of the real answer (neither key ever exists), so the hint fired unconditionally.
  Confirmed live: `cbm-trace.sh isNotEmpty in 1` against `RuoYi-Vue-Plus`'s `StringUtils.isNotEmpty` (a high-fanin utility method) returned 83 real callers in `.callers` — and the script's own `hint` field still read `"0 paths — the name must be EXACT: run cbm-find.sh first and copy the exact name"`, directly contradicting the 83 real records sitting right next to it in the same JSON object.
  This is a 100%-reproduction-rate bug (it fires on every single call, not a truncation-dependent edge case) — arguably more dangerous than a silent cap, since it's an actively wrong signal rather than an absent one. It did not, however, corrupt any past finding's numeric conclusions: every prior caller count documented in this file was hand-read off the raw `.callers`/`.callees` array by whoever ran the query, not taken from the (broken) hint text — confirmed by re-reading how each finding above was phrased, none of them quote or rely on this script's `hint` field as evidence.
  Fixed: the hint now checks `((.callers // []) + (.callees // []))|length`, which is correct regardless of direction.
- **`cbm-trace.sh` never set `include_tests`, so it silently used `trace_path`'s documented default of `false` (test-file callers/callees excluded) — a structural risk for any recall comparison against a grep ground truth that counts test files too.**
  `trace_path --help`: `--include-tests <boolean> Include test files in results. When false (default), test files are filtered out. When true, test nodes are included with is_test=true marker.`
  This could not be reproduced with a live divergent number on either currently-indexed dogfood repo — `RuoYi-Vue-Plus`'s only 4 real JUnit test files (`AssertUnitTest`/`DemoUnitTest`/`ParamUnitTest`/`TagUnitTest`, all under 75 lines) call nothing but `Assertions.*`/`System.out`/enum values, none of which trace back into indexed business logic; `plus-ui` has no test files at all. So this is a confirmed mechanism (the CLI's own docs are unambiguous), not yet a confirmed live discrepancy — flag it as a methodology caveat, not a retracted number, and re-check it if a future dogfood repo actually has substantive test-file callers.
  Fixed defensively anyway: the script now accepts a 4th positional arg (`true`/`false`, default `false`) to pass through, and its hint calls this out explicitly whenever the default is in effect and the result isn't empty, specifically so a future recall comparison against test-inclusive grep ground truth doesn't get misattributed to the graph tool.
- **The shared `_project.sh::cbm_call()` helper had no safety net for tools that fail with their error JSON on stderr instead of stdout — `set -e` would silently kill the whole calling script with zero output.**
  This exact gap was already flagged as a known followup in the `cbm-cypher.sh` section above (item 6: "the shared `_project.sh::cbm_call` itself is unchanged... those 6 scripts still have no equivalent protection") but never actually fixed — this pass closed it with a live repro rather than leaving it as a re-flagged TODO.
  Confirmed live: `trace_path` on an unknown function name exits with status 1, prints NOTHING to stdout, and writes a perfectly well-formed `{"error":"function not found","function_name":"...","hint":"Use search_graph(name_pattern=\"...\") to find the exact name, then pass it to trace_path."}` to stderr (after an unrelated `level=info msg=mem.init...` log line). `cbm_call()` only ever captured stdout via `$(...)`, so under the calling script's `set -euo pipefail`, the nonzero exit aborted the whole script at that line — the CLI's own genuinely useful error+hint was thrown away and the caller saw literally nothing, not even a hint to check the name.
  Fixed: `cbm_call()` now mirrors `cg_call()`'s (`scripts/_gate.sh`) tempfile + `jq empty` pattern — captures stderr separately, and if stdout isn't valid non-empty JSON, tries to recover the CLI's own well-formed error object from the last line of stderr before falling back to a generic `{"error","hint"}` wrapper. This is now the default for every `cbm-*.sh` script that calls `cbm_call()` (not just `cbm-cypher.sh`'s script-local `run_query()`, which already had its own equivalent).
  Side finding while implementing this: `jq empty` on a completely empty string exits **0** (success), not an error — meaning the naive "did stdout parse as JSON" check would have silently treated a genuinely empty, failed call as a valid blank success. Both `cbm_call()` and `cg_call()` (`_gate.sh`) now require stdout to be non-empty AND parse as JSON before treating it as a real answer; `cg_call()` had the same latent gap and is fixed alongside this for consistency, though no live repro of it misfiring on the codegraph side was found (codegraph's documented "not found" failure mode is always non-empty text on stdout with exit 0, not an empty-stdout/nonzero-exit case).
- **`cbm-impact.sh` and `cbm-arch.sh` both referenced output fields that don't exist in the real API response — always null, indistinguishable from "this repo genuinely has none of this."**
  `detect_changes`'s real response keys, confirmed live: `changed_files`, `changed_count`, `depth`, `impacted_symbols` — no `summary`, no per-symbol `risk` (this endpoint has no `--risk-labels` flag at all, unlike `trace_path`, which does). `cbm-impact.sh` asked for `.summary`/`.risk` anyway, so both were always `null`.
  `get_architecture`'s real response keys, confirmed live: `boundaries`, `clusters`, `edge_types`, `file_tree`, `hotspots`, `languages`, `layers`, `node_labels`, `packages`, `project`, `routes`, `total_edges`, `total_nodes` — no `entry_points` at all. `cbm-arch.sh` asked for `.entry_points[:10]` anyway, so it was always `null`.
  Neither dead field ever produced a wrong documented conclusion (nothing in this file or the benchmark cites either), but both looked exactly like "this feature exists but returns nothing" rather than "this field was never real" — worth fixing on sight regardless of whether it changed a past answer.
  Fixed: `cbm-impact.sh` now reports `changedFiles`/`changedCount`/`impacted` (real fields only). `cbm-arch.sh` drops `entry_points` and instead surfaces `layers`/`boundaries` — both real, both directly relevant to "architecture overview" questions, and both silently unavailable through this script before this fix (a user asking cbm-arch.sh for layer/boundary information got nothing, not because the tool lacks it, but because the wrapper never asked for it).
  Also confirmed live while checking this: `cbm-arch.sh`'s `routes` slice is a genuine overview cap from the underlying API itself, not something this wrapper introduces — `get_architecture` on `RuoYi-Vue-Plus` returned exactly 20 routes in `.routes` against a real total of 303 (cross-checked via `cg-find.sh -k route` and `cbm-cypher.sh routes`), with no `total`/`has_more` signal on this endpoint to detect it. `cbm-arch.sh` now says so explicitly in its own `hint` field — treat any `cbm-arch.sh` array field as a representative slice for an overview question, never as the answer to an exhaustive-listing question (which the Decision table already routes elsewhere).

## `cbm-*.sh`/`codegraph` parity audit — 2 more real defects fixed, plus a dead-end investigation worth recording so it isn't re-attempted (field-verified 2026-07-19)

Prompted by a direct ask to give `codebase-memory-mcp`'s scripts the same scrutiny the `codegraph` side had already had. Re-diffed every `cbm-*.sh` against a fresh `--help` dump of its underlying tool, one more time, on the theory that a defect class fixed once in one script can resurface in a sibling that was written earlier or touched less recently.

- **`cbm-grep.sh` had the exact silent-cap defect already fixed once in `cbm-find.sh`, unfixed.**
  `search_code --help` is explicit: `--limit <integer> Max enriched results per call. Default 10. Response includes 'total_grep_matches' and 'total_results' so callers can detect truncation. No offset parameter — raise limit or narrow with file_pattern / path_filter to see more.`
  The script hardcoded `limit:20` with no override at all, and passed through whatever `cbm_call` returned with zero post-processing — no error-object handling, no truncation hint, both `total_grep_matches`/`total_results` silently discarded.
  Confirmed live on `RuoYi-Vue-Plus`: `cbm-grep.sh 'public class'` returned exactly 20 results while `total_results` was **480** (`total_grep_matches`: 500 raw hits pre-dedup) — 460 real matches invisible with no signal.
  Fixed: added a `-n LIMIT` override (validated via `_require_positive_int`, default unchanged at 20), reshaped the output into the same `{count, results, hint}` shape every other `cbm-*.sh` script uses, and the hint now fires whenever `total_results` exceeds the returned count, naming the real total and pointing at `-n <higher>` — the CLI's own guidance, since `search_code` has no offset param to page with instead.
  Re-verified after the fix: `-n 500` on the same query returns all 480 with `hint: null`; a rare pattern with no truncation also correctly returns `hint: null`; a bad `-n` value fails clean with the same `_require_positive_int` error shape used elsewhere.
- **`cbm-cypher.sh`'s local `run_query()` had quietly become dead, duplicate validation logic — and a stale copy of the exact bug `_json_safe.sh` exists to prevent.**
  This script's own `run_query()` wrapper (see the item-6 entry above) was written *before* `_project.sh::cbm_call()` was hardened to guarantee valid JSON on every call (that followup is documented in the wrapper-script-defects section above). Once `cbm_call()` started guaranteeing this, `run_query()`'s own tempfile + status-check + `jq empty` re-validation on top of it became a no-op on the happy path — but it never got revisited, so it kept a bare `jq empty` with **no non-empty guard**, i.e. exactly the bug `_json_safe.sh`'s `_is_valid_json_answer()` was centralized specifically to stop from recurring (see that file's own comment: "this exact fix had to be applied twice... because the logic was duplicated instead of shared"). It had recurred a third time, just not yet triggered by a live empty-string case.
  Fixed: `run_query()` now only builds the request body and calls `cbm_call()` directly, keeping just the Cypher-specific error hint layered on top via `.+ {hint: ...}` — matching how every other `cbm-*.sh`/`cg-*.sh` script relies on the shared helper for the JSON guarantee instead of re-implementing it locally.
  Regression-verified: `hubs`/`routes` (200/303 truncation warning) /`cross-layer` (quote-injection-safe arg) all produce identical output to before; a deliberately malformed Cypher string now correctly produces `{error, exitCode, hint: "Cypher template hit a parser/engine error..."}` through the simplified path, confirming the error branch wasn't accidentally lost along with the dead validation code.
- **Dead-end investigated and closed: `index_status`'s `git.base_sha` is NOT a "last-indexed commit" staleness marker — do not build a stale-index check on it.**
  `_gate.sh` (codegraph side) warns when `codegraph status`'s `pendingChanges` shows added/modified/removed files pending a `sync` — `_project.sh` (cbm side) has no equivalent warning at all, which looked like a real parity gap. `index_status`'s `git` object has both `head_sha` and `base_sha`, and on `RuoYi-Vue-Plus` they were observed equal — plausible-looking evidence for "`base_sha` = commit at index time, compare against live `head_sha` to detect drift."
  Built a disposable throwaway git repo (outside any real project, deleted afterward) specifically to test this without touching the real indexed `RuoYi-Vue-Plus`/`plus-ui` projects, and could not reproduce the hypothesis under any of: a fresh index right after commit, switching branches, a genuinely diverged feature branch, or adding a local bare-repo remote with a set default branch — `base_sha` stayed the empty string `""` in every single case, never taking on a different value from `head_sha` at all.
  There is also no other field anywhere in `index_status`'s or `list_projects`'s response (checked both) recording anything like "commit sha when this project was indexed" — `nodes`/`edges`/`status`/`root_path`/`git.*` is the complete set.
  Conclusion: `base_sha` most plausibly reflects something tied to a real hosted-git-provider default-branch lookup (only ever observed non-empty on a repo cloned from an actual `github.com` remote) rather than an index-freshness marker, and it is currently impossible to detect "index built at an older commit than the working tree's current HEAD" through this CLI's exposed API at all — this is an **upstream capability gap**, not a `_project.sh` bug, and no fix was made. Re-check this only if a future `codebase-memory-mcp` release documents `base_sha`'s actual meaning or adds an explicit staleness/`indexed_at` field.

## Full regression pass after several rounds of fixes — the missing-arg-guard fix hadn't actually reached every script it applied to (field-verified 2026-07-19)

Prompted by a direct ask to re-verify this skill's own conclusions after "multiple rounds of bug fixes," on the theory (borne out below) that a fix applied to one script in a family is not evidence it was applied to every sibling with the identical pattern.

- **The missing-positional-arg guard added earlier this same day to `cbm-grep.sh`/`cbm-find.sh`/`cbm-snippet.sh` had NOT been extended to three more scripts with the identical `NAME="$1"`-with-no-guard pattern: `cbm-trace.sh`, and (codegraph side) `cg-impact.sh`/`cg-trace.sh`.**
  Confirmed live, all three: calling with zero positional args hit `set -u`'s unbound-variable trap and killed the script immediately — exit 1, completely empty stdout, not even a hint — before `cbm_call()`/`cg_call()` ever got a chance to run. This is the exact "total silent abort" failure shape already documented as the worse of the two variants when `cbm-find.sh`'s original bug was first found and fixed.
  Fixed: all three now guard with `[ "$#" -ge 1 ] || { echo '{"error":...,"hint":...}'; exit 2; }` before referencing the positional arg, matching the other fixed scripts. Regression-verified: `cbm-trace.sh isNotEmpty in 1` still returns 83 callers (matching the number on record); `cg-trace.sh`'s Facade/`useFlash`/`hi.events` numbers (75 records/28 files, 30/30, 43) all still reproduce exactly on the still-indexed `pterodactyl-panel-bench`/`hievents-bench` codegraph clones after the fix, confirming the guard clause didn't disturb the bridging/truncation logic it now sits in front of.
- **`cbm-cypher.sh` itself — fixed and committed earlier THIS SAME DAY — still had this exact bug, because the fix round only tested an unknown-template argument, never a fully-missing one.**
  `case "$1" in ...)` at the top of the template dispatch references `$1` directly; confirmed live, zero args crashes identically to the plain-assignment cases above (`line 101: $1: unbound variable`, empty stdout, exit 1). A cross-check against "did we test every no-args case, not just every bad-args case" would have caught this in the original fix round; it didn't, which is itself the reason a dedicated regression pass is worth doing after a batch of fixes rather than assuming a reviewed diff is a fully-tested one.
  Fixed the same way, regression-verified against `hubs` (unchanged top-3: StringUtils/R/LoginHelper) and the zero-arg case (now a clean `{"error","hint"}`, exit 2).
- **`_require_positive_int` (`scripts/_json_safe.sh`), shared by six call sites across four scripts, wrote its error to STDERR ONLY with no `hint` field — leaving STDOUT completely empty on a bad numeric arg.**
  This is the identical failure shape `_is_valid_json_answer()` exists to catch downstream (`jq empty` on a genuinely empty string exits 0, so a caller that only captures stdout and pipes it through `jq empty` would see a false "valid blank success" instead of a structured error) — just reintroduced one level up, at the one arg-validation helper every numeric-flag script in this skill shares (`cbm-grep.sh -n`, `cbm-find.sh -n/-o`, `cbm-trace.sh`'s depth, `cg-trace.sh`'s limit).
  Fixed: now prints its error JSON (with a `hint`) to STDOUT, matching every other error path in this skill. Confirmed live via `cg-trace.sh <symbol> both abc` (an invalid limit): stdout now carries `{"error":"invalid value for limit (3rd arg): 'abc' is not a positive integer","hint":"pass a positive integer for limit (3rd arg)"}`, exit 2 — previously stdout was empty and only stderr had the (hint-less) error.
- **Full happy-path smoke test across all 14 `cbm-*.sh`/`cg-*.sh` scripts, both dogfood repos, confirms no other regression from any fix applied this session or in prior rounds.** Every headline number already on record reproduced exactly: `RuoYi-Vue-Plus` — `routes` 303 (`cg-find.sh -k route`), `hubs` top-3 unchanged (StringUtils 279/R 259/LoginHelper 230), `isNotEmpty` 83 callers at depth 1; `plus-ui` — `hubs` empty (0 rows, confirmed still a real modeling gap not a regression), `cg-find.sh -k component` 99/99, `cbm-grep.sh` Chinese term found (13 total, matching the "byte-for-byte matches manual grep" claim in kind if not the exact query used originally); `pterodactyl-panel-bench`/`hievents-bench` (codegraph-only) — Facade 75 records/28 files, `useFlash` 30/30, `hi.events findById` 43, all `possiblyTruncated: false`. Nothing published before this pass needed a numeric retraction — this round's defects were all argument-validation/output-plumbing bugs on the *missing-input* path, not the normal-data path any existing finding was read off of.
- **Practical lesson, stated plainly since it's the reason this pass happened**: a fix verified against "a bad argument" and "the happy path" is not the same as a fix verified against "no argument at all" — the three scripts above, plus `cbm-cypher.sh`'s own same-day regression, all slipped through specifically because the zero-args case wasn't part of the original test matrix. Treat "test the missing-arg case explicitly, not just the bad-arg case" as a standing item whenever a positional-arg guard is added to a new script in this family.

## `cg-node.sh`'s caller/callee trail truncation is honest, unlike `callers`/`callees`'s old silent cap (field-verified 2026-07-20, `hi.events`)

While cross-checking `codegraph node --help` against `cg-node.sh`, a real asymmetry turned up: `codegraph node` has **no** `-l`/`--limit` flag at all in symbol mode (file mode's `--limit` caps *lines*, not call-trail entries) — so there is no way to ask this specific command for a longer trail the way `-l` fixes `callers`/`callees`.
This looked like it might be the same silent-cap bug as the one fixed in `cg-trace.sh`, but it isn't: `codegraph node`'s own markdown output ends a cut-off "Called by ←" list with an explicit `+N more` marker.
Confirmed exact on `hi.events` (still-indexed from Round 3): querying `findById` returned, among its 6 definitions, `RepositoryInterface.php:95`'s "Called by ←" line with 12 callers listed followed by `+31 more` — 12 + 31 = **43**, matching the already-verified real caller count for this same symbol family (`EventRepositoryInterface::findById`, established via `cg-trace.sh -l 200` in Round 3).
Conclusion: `cg-node.sh`'s trail is a legitimate, self-reporting preview — treat a trail ending in `+N more` as a real signal to escalate to `cg-trace.sh` with a higher limit for the complete list, but do NOT treat a short trail with no such marker as evidence of hidden truncation; that failure mode belongs to `callers`/`callees` before the 2026-07-19 fix, not to `node`.
